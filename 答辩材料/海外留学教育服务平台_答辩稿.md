## 第 1 页｜把分散的留学服务，连接成一个可追踪的业务闭环

- 主讲人：曾铁昭
- 建议用时：30 秒

各位老师、同学大家好，我们是海外留学教育服务平台项目组，我是曾铁昭，今天和龙梦婷共同完成项目答辩。我们的项目不是单独做一个聊天机器人，而是希望把留学咨询从潜在客户识别、客服咨询、顾问跟进、学生服务到管理报告连接起来，让不同角色在同一个平台中协同。接下来我们会重点介绍五个业务模块、它们之间的关系，以及开发过程中真实做过的技术取舍。

## 第 2 页｜问题不在缺少工具，而在信息和流程彼此割裂

- 主讲人：曾铁昭
- 建议用时：45 秒

传统留学服务的难点主要有三个。第一，客户资料、咨询记录和跟进信息散落在表格、聊天记录和不同人员手里，容易重复录入。第二，从咨询到签约再到在读服务，流程跨度很长，状态变化难追踪。第三，管理者看得到很多记录，却很难快速形成可信的经营和服务报告。因此我们把项目目标定义成一条完整链路：客户资料进入系统后能够被研判、承接、跟进和服务，最终沉淀为可分析的数据，而不是做五个互不关联的功能页面。

切页提示：下一页展示五个模块如何共同完成这条链路。

## 第 3 页｜五个模块共同覆盖客户全生命周期

- 主讲人：曾铁昭
- 建议用时：50 秒

这张图是整个项目最重要的业务关系。客户研判负责把原始资料变成结构化画像和匹配结果；客服 Agent 承接咨询、课程、活动和报名；企业智能助手把意向客户、跟进记录和员工日报组织起来；学生智能助手负责签约后的请假、成绩、通知、投诉和服务咨询；智能报告再从这些业务数据中生成报告、周期比较和管理行动。它们不是简单的前后端页面集合，而是一条从获客、转化、交付到管理反馈的闭环。后面的每个模块页都会说明自己在这条链路中的输入、处理和输出。

切页提示：交给龙梦婷介绍支撑五模块的技术架构。

## 第 4 页｜统一平台采用“纯 Python 为主、客服保留 Dify”的混合架构

- 主讲人：龙梦婷
- 建议用时：50 秒

技术上，前端使用 React、TypeScript 和 Vite，统一通过 API Client 调用 FastAPI。后端按照 Router、Schema、Service、Model 分层，业务数据通过 SQLAlchemy 访问数据库；需要大模型能力时，由 Python 服务完成 Prompt、结构化输出和异常控制。这里有一个必须讲清楚的边界：只有曾铁昭负责的客服 Agent 使用 Dify 加 Python，其他四个业务模块都是纯 Python 实现。这样既保留了 Dify 在客服对话编排上的优势，又避免把所有模块都绑定到工作流平台。

切页提示：从第一个模块——客户研判开始。

## 第 5 页｜客户研判把原始资料变成可跟进的客户画像

- 主讲人：龙梦婷
- 建议用时：55 秒

客户研判模块解决的是“资料很多，但顾问不知道先跟谁、适合什么方案”的问题。输入可以是客户基础信息和资料内容，后端通过 `routers/profile.py` 接收请求，在 `profile_service.py` 中完成字段处理、规则匹配和画像组织，再把结果保存到客户画像和规则相关模型。输出不仅是一个判断结论，还要包含结构化画像、匹配依据和后续行动信息。它的下游是 CRM：研判结果可以转成意向客户，继续进入客服咨询和顾问跟进。当前模块采用纯 Python，避免把文件处理和业务规则放进 Dify。

切页提示：交回曾铁昭，介绍唯一使用 Dify 的客服 Agent。

## 第 6 页｜客服 Agent 用对话串联课程、活动、报名和会话记录

- 主讲人：曾铁昭
- 建议用时：60 秒

客服 Agent 面向潜在客户和在服用户，负责高频、标准化的咨询入口。用户提问后，Dify 负责对话流程和意图编排，Python 后端负责真正的课程查询、活动查询、报名校验以及会话消息落库。相关代码主要位于 `routers/chat.py`、`services/chat_service.py` 和聊天模型中。我们把职责分开，是为了防止工作流直接承担数据库规则：例如重复报名、参数校验和鉴权必须由后端控制。客服 Agent 的上游可以接收客户研判结果，下游可以把咨询意向和报名结果交给 CRM 或后续学生服务。

演示提示：若 Dify 环境可用，可演示“查询近期活动—继续报名”；若环境不可用，只演示 Python 接口并明确说明在线环境未验收。

## 第 7 页｜企业智能助手把客户跟进和团队协作沉淀为业务事实

- 主讲人：曾铁昭
- 建议用时：50 秒

企业智能助手服务顾问、运营和管理人员。它把 CRM 客户、跟进记录、员工日报和组织查询放到统一工作台中。请求从 `routers/crm.py` 和 `routers/assistant.py` 进入，由对应 Service 完成查询、状态更新和业务校验。这个模块的重点不是让大模型随意生成 SQL，而是让自然语言意图映射到受控的只读查询和明确的写操作。它从客户研判和客服模块接收客户及咨询信息，同时把真实跟进数据提供给智能报告，也为学生问题升级和跨角色协同提供处理入口。

切页提示：交给龙梦婷讲签约后的学生服务链路。

## 第 8 页｜学生智能助手覆盖签约后的服务、提醒和人工介入

- 主讲人：龙梦婷
- 建议用时：60 秒

学生智能助手面向签约学生，解决服务事项多、状态不透明和问题容易遗漏的问题。模块覆盖请假、成绩、通知、投诉、心理记录、申请进度和多轮对话。接口主要位于 `routers/student.py` 和 `routers/student_chat.py`，业务规则集中在 `student_service.py`。例如请假不是简单保存一条数据，还包括提交、审批和状态更新；投诉和心理高风险场景需要明确人工介入，不能让模型替代人工判断。它的上游来自客服转化和 CRM 服务关系，下游会把处理状态、风险和服务结果汇总给企业工作台与智能报告。

## 第 9 页｜智能报告把业务记录转成有证据、可追踪的管理结论

- 主讲人：龙梦婷
- 建议用时：60 秒

智能报告模块面向管理者，不只是生成一段总结，而是把报告生成、列表、详情、计划、行动项和数据维护组织成完整流程。后端由报告 Router、`report_service.py` 以及 `services/reporting` 下的聚合器、规则、Prompt 和助手模块共同完成。报告助手支持周期比较和受控的跨报告分析，同时返回证据、数据质量和澄清信息。它的输入来自 CRM、学生服务和运营数据，输出则是报告内容、风险信号和可跟踪行动，重新反馈给企业管理和后续服务优化。

## 第 10 页｜模块之间通过业务对象交接，而不是彼此直接耦合

- 主讲人：曾铁昭、龙梦婷
- 建议用时：65 秒

曾铁昭：模块关系的前半段从客户资料开始。研判模块生成客户画像和匹配依据，客服模块承接咨询、课程、活动和报名，企业助手再把有效意向沉淀为 CRM 客户和跟进任务。这里交接的是客户、咨询、报名和跟进等业务对象，而不是一个模块直接调用另一个模块内部函数。

龙梦婷：签约以后，学生助手继续记录服务事项、状态和风险；智能报告读取这些业务事实，形成管理视角的汇总、比较和行动建议。管理结果再回到企业工作台，帮助调整跟进和服务。这样每个模块可以独立测试，也能够通过统一数据库口径和 API 契约协同。

## 第 11 页｜一条请求经过接口、业务规则、数据与模型，再返回可解释结果

- 主讲人：曾铁昭、龙梦婷
- 建议用时：55 秒

曾铁昭：用户操作先从 React 页面进入统一 API Client，请求携带 Token 到 FastAPI Router。Router 只负责接收参数、依赖注入和返回响应，核心业务规则交给 Service，避免接口函数越来越臃肿。

龙梦婷：Service 根据场景访问 SQLAlchemy Model，必要时调用受控的大模型客户端，然后把结构化结果返回前端。整条链路还要经过权限、参数校验、异常转换、日志和数据来源标记。我们希望做到的不只是“模型能回答”，而是每一步都知道输入来自哪里、数据写到哪里、失败后先排查哪一层。

我们在工程上重点处理了五类问题：基于角色的访问控制、统一业务异常、报告生成幂等、真实与 Mock 数据边界，以及敏感配置隔离。本轮实际验证中，后端执行 507 项测试，503 项通过、4 项失败；失败集中在 Swagger 鉴权契约、种子脚本索引重名、直接启动 reload 配置和 OpenAPI 快照漂移。前端结构检查和 92 项操作目录检查通过，生产构建也通过。我们不会把这些结果包装成全绿，而是把未关闭项作为明确技术债。

## 第 13 页｜Vibe Coding 提高了速度，但不能替代契约、测试和人工判断

- 主讲人：龙梦婷
- 建议用时：60 秒

开发过程中我们使用了 Vibe Coding，包括让 AI 协助拆任务、生成代码草稿、补测试和分析报错。它最大的价值是提高起步速度，但也带来了几个坑。第一，多人和 AI 同时修改 `main.py`、配置和公共模型时，出现过路由被覆盖和导入冲突。第二，AI 生成的代码局部看起来正确，但可能漏掉公共配置或让 OpenAPI 快照漂移。第三，最初所有模块都计划使用 Dify，实际联调后发现成本太高，所以第二天晚上调整为只有客服保留 Dify。我们的经验是：AI 负责加速，人必须负责冻结契约、控制修改范围、看 diff、跑测试，并能解释每一行关键代码为什么存在。

## 第 14 页｜我们交付的不只是五个功能，而是一条可继续演进的服务链路

- 主讲人：龙梦婷
- 建议用时：40 秒

最后总结一下。我们完成了客户研判、客服 Agent、企业智能助手、学生智能助手和智能报告五个重点模块，并通过统一前端、FastAPI 分层和数据库契约把它们连接起来。项目目前是教学与演示级交付，不等同于生产系统；真实外部环境、四项工程问题以及更完整的 RAG、缓存和异步任务仍是后续方向。这个项目让我们真正练习了从业务需求、接口和数据库，到 AI 能力、联调、测试和交付复盘的完整过程。我们的介绍到这里，谢谢各位老师和同学，下面进入演示或提问环节。

## 现场演示建议

优先选择两条稳定链路，控制在 2—3 分钟：

1. 客服：课程或活动查询 → 报名；
2. 学生：请假提交 → 状态查询，或成绩/通知查询；
3. 若时间允许，再展示智能报告列表 → 详情 → 助手追问。

现场演示失败时，不要反复点击。直接切换到 Swagger、前端截图或已有接口结果，并说明失败发生在前端、接口、数据库还是外部服务层。

## 高频追问速答

### 为什么只有客服使用 Dify？

客服对话流程相对标准，适合用 Dify 做意图编排和对话管理；其他模块包含较多数据库规则、状态流转和可测试的业务逻辑，用纯 Python 更容易控制边界、复用 Service 并进行自动化测试。

### 五个模块之间怎么解耦？

模块通过统一业务对象、数据库口径和 API 契约交接，不直接依赖对方内部函数。每个模块保留独立 Router、Service、Schema 和 Model 职责。

### 大模型回答错误怎么办？

高风险业务不让模型直接做最终决定；使用结构化输出、受控工具、证据和数据质量字段，并在政策、心理和写操作场景保留人工确认。

### 当前最大的不足是什么？

真实外部环境验收还不够完整，后端仍有 4 项已知测试失败；此外生产部署、缓存、异步任务和完整 RAG 不属于本轮已完成范围。
