# 项目 Agent 协作规则

本文件是 `education-service-api` 项目的根级协作规则。所有 Agent、AI 编码助手和协作者在本项目中新增或修改代码时，都必须优先遵守这里的要求。

## 1. 适用范围

- 本规则约束以后新增或修改的代码，不要求一次性重写历史旧代码。
- 如果后续修改到旧文件，必须让被修改的接口、类、函数、关键逻辑块同步符合本规则。
- 本项目是教学型、就业导向型 FastAPI 后端项目，代码不仅要能运行，还要能帮助其他同学看懂、复盘、排错和面试表达。
- 变量名、函数名、类名、字段名继续使用英文；解释说明、docstring、关键注释必须使用清晰中文。

### 1.1 模块技术栈边界

- 只有客服 Agent 模块使用 `Dify + Python`：Dify 负责客服 Agent 的工作流编排，Python 负责接口接入、自定义业务逻辑、数据处理和外部服务集成。
- 除客服 Agent 模块外，其他所有业务模块均使用纯 Python 实现，不引入 Dify 工作流、Dify 应用或 Dify 依赖。
- 新增模块或修改现有模块前，必须先判断它是否属于客服 Agent。只有明确属于客服 Agent 的代码，才允许接入 Dify；边界不清楚时，不得默认使用 Dify。
- 进行架构设计、代码实现、注释、测试、项目复盘和面试表达时，都必须保持上述技术栈边界一致，避免把 Dify 描述为整个项目的通用技术底座。

## 2. Superpowers 使用规则

框架搭建、新模块、CRUD、接口开发、业务逻辑、排错、重构，都必须先使用 Superpowers 相关 skill，再进入代码修改。

- 新功能、新模块、框架搭建：先使用 `superpowers:brainstorming` 明确目标和设计，再使用 `superpowers:writing-plans` 写出可执行计划。
- CRUD、接口、业务代码：必须使用 `superpowers:test-driven-development`，先明确行为和验证方式，再写实现。
- 报错、接口异常、测试失败、数据不符合预期：必须使用 `superpowers:systematic-debugging`，先定位现象和根因，再修改代码。
- 执行已有计划：使用 `superpowers:subagent-driven-development` 或 `superpowers:executing-plans`，按计划逐步实现和验证。
- 完成前：必须使用 `superpowers:verification-before-completion`，用实际命令、diff 或检查结果证明任务完成。

如果用户明确要求跳过某个流程，仍然要优先说明风险；除非用户坚持，否则不要绕过上述流程。

## 3. 注释生成四段法

本项目注释参考 `D:\个人\！！！沃林学习\个人项目\思维框架\communication-response-system-chat-copy` 的四段结构：判断维度、注释逻辑、注释框架、表达细节。

写注释前不要只看代码表面在做什么，而要先判断“谁会读这段代码、他可能卡在哪里、这段代码在项目链路里承担什么责任”。注释的目标不是把代码逐字翻译成中文，而是让读者知道为什么这样写、数据怎么流动、出错时先查哪里。

### 3.1 判断维度：先判断这段代码需要解释什么

新增或修改代码时，先按以下维度判断注释深度。

- 代码层级：判断当前代码属于文件、路由、Service、Schema、Model、工具函数、测试，还是脚本入口。
- 使用场景：说明它服务于接口请求、数据库读写、权限校验、事务提交、分页筛选、文件处理、AI/RAG 链路、外部服务调用中的哪一环。
- 读者卡点：考虑读者可能是概念不懂、参数不懂、数据流不懂、分层职责不懂，还是排错时不知道看哪里。
- 信息深度：按“能看懂、能复述、能排错、能面试讲清楚”的教学项目标准写。
- 行动出口：读完注释后，读者应该知道这段代码怎么调用、怎么验证、出错后先查哪一层。

### 3.2 注释逻辑：决定说明顺序

注释内容必须按读者容易接住的顺序组织。

- 先说“这段代码解决什么问题”，再说“它是什么”。
- 先说“数据从哪里来”，再说“中间怎么处理”，最后说“结果返回给谁或写到哪里”。
- 先说明正常流程，再说明异常、边界、权限和风险。
- 对关键调用必须说明为什么这样写，而不是只说调用了什么。
- 如果某段代码是为了避免问题，要补一句“不这样会怎样”。

关键调用示例：

- `Depends(get_db)`：说明这是 FastAPI 在接口执行前准备数据库会话，不是手动创建连接。
- `Depends(get_current_user)`：说明这是从请求 Token 中解析当前用户，用于权限判断和审计。
- `model_dump(exclude_unset=True)`：说明这是只取前端实际传入的字段，避免部分更新时覆盖未提交字段。
- `db.commit()`：说明这是把本次写操作提交到数据库；没有提交时，修改只停留在当前会话中。
- `db.refresh(obj)`：说明这是把数据库生成或更新后的字段重新读回 ORM 对象，例如主键、更新时间。
- `setattr(obj, field, value)`：说明这是根据请求字段动态更新对象，适合字段较多的部分更新接口。
- `raise HTTPException(...)`：说明这是把业务异常转换成前端能理解的 HTTP 状态码和错误信息。

### 3.3 注释框架：不同代码对象使用不同槽位

不同位置的注释必须覆盖不同信息槽位，不能所有地方都写成同一种空泛说明。

#### 文件注释

每个新增或重点修改的 Python 文件顶部必须写模块级 docstring，至少说明：

- 本文件负责哪个业务模块。
- 它位于项目哪一层，例如路由层、业务层、数据模型层、Schema 层、工具层、测试层。
- 上游是谁调用它，例如 FastAPI 路由、Service、测试用例、定时任务。
- 下游依赖什么，例如数据库 Session、ORM Model、Pydantic Schema、外部 API、Redis、MinIO、Milvus。
- 这个文件中最重要的接口、类或函数是什么。

#### 接口注释

每个 FastAPI 路由函数必须写中文 docstring，至少说明：

- 这个接口解决什么业务问题。
- 请求来自哪里，典型使用者是谁，例如前端页面、管理端、定时任务、第三方回调。
- 核心路径参数、查询参数、请求体字段分别影响什么。
- 是否需要登录、角色权限或数据权限。
- 会调用哪个 Service 或核心函数。
- 返回给前端的关键字段是什么。
- 常见异常是什么，例如未登录、无权限、数据不存在、参数非法、外部服务失败。

#### 类注释

每个 Pydantic Schema、SQLAlchemy Model、业务类必须写中文 docstring，至少说明：

- 这个类代表什么业务对象。
- 如果是 Model，要说明对应哪张数据库表。
- 如果是 Schema，要说明它用于请求体、响应体、内部数据传递，还是配置对象。
- 关键字段在业务中的含义。
- 字段命名是否需要和数据库、前端或接口文档保持一致。

#### 函数注释

每个函数和方法必须写中文 docstring，至少说明：

- 业务目的：这个函数为什么存在。
- 参数含义：每个重要参数从哪里来、影响什么。
- 返回值：返回给谁、返回结构如何使用。
- 副作用：是否读库、写库、提交事务、调用外部服务、写文件、发消息。
- 异常：可能抛出什么业务异常或 HTTP 异常。
- 边界：哪些情况返回空列表、`None`、默认值或错误。

#### 对象和变量注释

关键对象创建或关键中间变量必须有中文注释，尤其是：

- 数据库会话，例如 `db: Session`。
- 当前登录用户，例如 `current_user`。
- ORM 查询对象，例如 `query`。
- 前端传入的部分更新字段，例如 `update_data`。
- 请求 payload、外部 API 响应、LLM 返回结果、Embedding 向量、检索结果。
- 分页参数、过滤条件、状态字段、逻辑删除字段。

#### 逻辑块注释

涉及以下逻辑时，代码块前必须写中文注释说明业务原因：

- `if` 权限判断、状态判断、参数边界判断。
- `for` 批量处理、字段遍历、结果组装。
- `try/except` 异常捕获和降级处理。
- 分页、排序、筛选、模糊搜索。
- 逻辑删除、状态流转、审批流、重试。
- 事务提交、回滚、刷新对象。
- 文件上传、文件解析、文本清洗、切分、Embedding、向量入库、检索召回、Prompt 拼接、大模型调用。

#### 返回值注释

接口返回 `dict`、Schema、列表、分页结果时，要说明：

- 哪些字段给前端展示。
- 哪些字段给下游继续处理。
- 哪些字段用于排查问题，例如任务 ID、状态、错误信息。
- 如果返回分页结构，要说明 `items`、`total`、`page`、`page_size` 的含义。

#### 测试注释

新增测试用例要写清楚：

- 测试目标：验证哪个接口、函数或业务规则。
- 准备数据：为什么要准备这些数据。
- 执行动作：调用哪个函数或接口。
- 断言含义：为什么这个断言能证明行为正确。

### 3.4 表达细节：注释要具体、动作化、有边界

中文注释要让人读完能行动，不要只写正确但空泛的话。

- 把抽象名词改成具体动作。
  - 不要只写“权限管理”。
  - 要写“判断当前用户能不能访问或修改这条数据，避免普通用户越权操作管理端数据”。
- 先讲它解决什么问题，再讲它是什么。
  - 不要只写“使用环境变量读取配置”。
  - 要写“把数据库密码、JWT 密钥等敏感配置从代码中拆出来，避免提交代码时泄露；运行时再从环境变量读取”。
- 补一句“不这样会怎样”。
  - 例如“如果不使用 `exclude_unset=True`，前端没传的字段也可能被默认值覆盖，导致部分更新变成全量覆盖”。
- 给具体例子，不只讲原则。
  - 例如解释分页时，说明“第 2 页、每页 10 条，对应跳过前 10 条再取 10 条”。
- 给边界，不把话说死。
  - 多用“一般情况下”“除非接口要求全量覆盖”“这里用于教学项目，所以写得更详细”。
- 控制单句信息密度。
  - 一句话只承担一个主要说明任务；复杂逻辑拆成短句或列表。
- 把结论压缩成一句可复述的话。
  - 例如“Service 层负责业务规则，Router 层只负责接请求和返回响应”。

## 4. 注释质量边界

- 允许比真实企业项目更详细，因为本项目同时承担学习、协作、复盘和面试表达训练。
- 不写无信息量注释，例如“创建对象”“调用函数”“返回结果”。
- 不机械翻译代码，例如 `db.add(user)` 不要只写“添加用户”，要说明“把新用户对象加入当前数据库会话，等待后续 `commit` 一次性提交”。
- 不用拼音变量名；变量、函数、类名保持英文，注释解释用中文。
- 不为了注释而破坏代码可读性；简单自解释语句可以少写，但接口、数据库、权限、事务、状态、外部服务、AI/RAG 链路必须详细写。

## 5. FastAPI 分层注释重点

本项目采用 FastAPI、SQLAlchemy、Pydantic 的常见后端分层。后续新增或修改代码时，各层注释重点如下。

- `routers/`：解释接口目的、请求参数、依赖注入、权限校验、调用哪个 Service、返回什么。
- `services/`：解释业务规则、数据库读写、状态变化、事务边界、异常处理。
- `schemas/`：解释请求体和响应体的字段含义、校验规则、与前端或数据库字段的关系。
- `models/`：解释数据库表含义、关键字段、索引/唯一约束、逻辑删除或状态字段。
- `utils/`：解释工具函数解决什么通用问题、输入输出、失败时怎么处理。
- `tests/`：解释测试场景、输入数据、预期结果、为什么这个断言能证明规则正确。

## 6. AI/RAG 链路注释重点

后续涉及 AI 应用开发、知识库、RAG、Agent、文件解析、Embedding、向量库、缓存和对象存储时，注释必须体现完整链路位置。

其中，只有客服 Agent 链路允许使用 Dify 进行工作流编排；知识库、RAG、文件解析、Embedding、向量检索以及其他 Agent 或业务模块均由纯 Python 实现。相关注释必须明确当前模块采用哪种实现方式，不能把客服 Agent 的 Dify 方案套用到其他模块。

推荐在相关代码注释中说明它属于下面哪一环：

```text
用户输入
-> FastAPI 接口
-> 文件上传 / 文本接收
-> 文件存储 / MinIO
-> 文档解析 / OCR / ASR
-> 文本清洗 / 切分
-> Embedding
-> Milvus 向量入库
-> 检索召回
-> Prompt 拼接
-> 大模型调用
-> 结果返回
-> 日志 / 异常 / 缓存 / 权限 / 性能优化
```

注释要说明当前代码负责哪一步、输入来自哪里、输出给谁、失败时怎么排查。

## 7. 提交前自查清单

每次完成新增或修改代码后，至少检查以下问题：

- 是否按任务类型使用了对应 Superpowers skill。
- 新增或修改的文件是否有中文模块说明。
- 新增或修改的接口、类、函数是否有中文 docstring。
- 关键对象、关键调用、关键逻辑块是否解释了“为什么这样写”。
- 是否说明了数据来源、处理过程、输出去向。
- 是否说明了权限、异常、事务、边界和排错入口。
- 是否存在“创建对象”“返回结果”这类无信息量注释，如果有就改成业务解释。
- 是否运行了和本次修改相匹配的验证命令。
