  Iteration 1.1 完成 — 异步契约与运行可靠性修正

  1. 同步报告生成问题的修复方式

  问题：tools.py 中 tool_generate_existing_report() 在 HTTP 请求线程内同步调用 generate_report(report.id, db)，阻塞请求线程 3-10s。

  修复：
  - 从 tools.py 移除 from services.reporting.orchestrator import generate_report 和 generate_report(report.id, db) 调用
  - 工具只创建任务记录并返回，不做任何耗时操作
  - 在 service.py 的 handle_message() 中，任务创建成功后通过 background_tasks.add_task(generate_report_async, report.id) 注册后台执行
  - generate_report_async 在独立 Session 中执行，不占用请求线程

  2. 智能入口完整请求流

  POST /api/v1/reports/assistant/messages
  → REPORT_ASSISTANT_ENABLED 检查（503 if disabled）
  → ensure_management_user（403 if student）
  → ReportAssistantService.handle_message()
    → Step 1: build_report_catalog（按角色过滤）
    → Step 2: IntentParser.parse()（LLM 或关键词降级）
    → Step 3: decide_clarification()（置信度 + 权限）
    → Step 4: resolve_assistant_period()（时间解析）
    → Step 5: _execute_tool() → tool_generate_existing_report()
      → create_report_task(idempotency_key=...)
    → Step 5b: background_tasks.add_task(generate_report_async, report_id)
    → Step 6-7: 组装响应（answer + context + evidence）
  → 返回 200（含 status="generating" + report_id）
  → 后台：generate_report_async() 在独立 Session 中执行

  3. 是否真正复用了原 BackgroundTasks

  是。智能接口与 POST /api/v1/reports/generate 使用完全相同的后台执行链路：
  - 相同的 generate_report_async(report_id) 入口
  - 相同的独立 SessionLocal() 创建模式
  - 相同的 generate_report(report_id, db) 执行逻辑
  - 相同的 Registry → Aggregator → AI Generator → Renderer 编排链

  4. 幂等键完整传递链路

  client_request_id（请求体字段）
  → service.py: f"manual:{request.client_request_id}"
  → _execute_tool(idempotency_key=...)
  → tool_generate_existing_report(idempotency_key=...)
  → create_report_task(idempotency_key=...)
  → build_idempotency_key(request_key=...)
  → report_generation.idempotency_key = "manual:{key}"
  → DB UNIQUE INDEX 保证幂等
