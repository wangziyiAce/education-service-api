# 智能报告模块集成联调说明

## 1. 交付边界

本阶段冻结 `master` 当前智能报告能力，供团队合并和统一前端联调使用。智能报告由纯 Python 实现，不接入客服 Agent 的 Dify 工作流。本阶段不新增 Iteration 4、Redis、Celery、RAG、多 Agent、Docker 或生产部署配置。

真实接口以同目录 `openapi-iteration3.json` 为准；该文件必须通过 `python scripts/export_report_openapi.py` 从 `main.app` 重新生成，禁止手写。

## 2. 请求链路

```text
前端登录并保存 JWT
→ Axios 使用 /api/v1 并注入 Authorization: Bearer <token>
→ 获取 GET /reports/types
→ POST /reports/generate 创建异步任务
→ 202 返回报告 ID 和 pending/generating 状态
→ GET /reports/{report_id} 轮询详情
→ completed 后展示 report_content、report_html、data_quality
→ POST /reports/assistant/messages 继续追问
→ 前端把响应 conversation_context 原样传入下一轮
```

报告状态没有单独的 `/status` 路由，前端使用 `GET /api/v1/reports/{report_id}` 查询任务状态。前端 API Client 已配置 `/api/v1`，因此业务模块内使用 `/reports/...` 相对路径。

## 3. 冻结接口组

| 能力 | 方法与路径 | 成功状态 | 关键说明 |
|---|---|---:|---|
| 报告类型 | `GET /api/v1/reports/types` | 200 | 返回 10 类报告、角色、过滤字段和 JSON Schema |
| 创建报告 | `POST /api/v1/reports/generate` | 202 | 使用 `Idempotency-Key` 请求头防止重复创建 |
| 报告列表 | `GET /api/v1/reports` | 200 | `items/total/page/page_size` 分页结构 |
| 报告详情与状态 | `GET /api/v1/reports/{report_id}` | 200 | 不存在 404；普通员工只能访问自己生成的报告 |
| 失败重试 | `POST /api/v1/reports/{report_id}/retry` | 202 | 创建新记录，不覆盖原失败报告 |
| 报告行动项 | `/api/v1/reports/{report_id}/actions` | 200 | POST 确认行动、GET 查询行动 |
| 更新行动项 | `PATCH /api/v1/reports/actions/{action_id}` | 200 | 只更新前端实际提交字段 |
| 行动兼容接口 | `/api/v1/report-actions/{action_id}` | 200 | GET/PATCH，权限继承关联报告 |
| 报告计划 | `/api/v1/report-schedules` | 200/201 | 创建、列表、详情、修改、删除 |
| 报告数据 | `/api/v1/report-data/*` | 200/201 | 申请材料、渠道成本、合同、回款 |
| 智能报告助手 | `POST /api/v1/reports/assistant/messages` | 200/202 | 生成任务返回 202，其余按响应状态映射 |

## 4. 核心请求契约

报告生成请求至少包含 `report_type` 和 `report_title`，周期与 `filters` 可选。相同业务请求应复用同一个 `Idempotency-Key`；网络重试时不能生成新键。

智能助手请求：

```json
{
  "message": "对比本周和上周的申请风险",
  "conversation_context": {
    "conversation_id": "由前端生成并持续复用"
  },
  "client_request_id": "同一次提交保持不变"
}
```

前端必须识别响应的 `status`、`intent`、`report_id`、`report_type`、`answer`、`assumptions`、`evidence`、`data_quality`、`comparison_period`、`metric_comparisons`、`comparison_data_quality`、`relationship_sections`、`suggested_follow_ups` 和 `conversation_context`。Evidence 和比较结果只能展示后端返回内容，前端不得重新计算指标、拼接因果结论或重建上下文。

## 5. 权限与错误处理

- 所有报告接口需要有效 JWT；未登录或 Token 失效返回 401。
- 学生禁止访问管理报告、报告类型、报告计划和智能报告助手，返回 403。
- 管理员、经理、组长可访问管理范围报告；普通员工只能访问自己生成的报告。
- 参数或报告类型非法返回 400/422；资源不存在返回 404。
- 智能助手关闭时返回 503；助手内部 `status=error` 时路由映射为 500。
- 401 清理本地 Token 并跳转登录；403 保留当前页面并提示权限不足；500/503 应允许用户重试，不能伪造成功结果。

## 6. 配置、数据库与合并入口

报告内容真实调用使用 `REPORT_AI_MODE=llm` 和 `REPORT_LLM_*`。助手还需要开启 `REPORT_ASSISTANT_ENABLED=true` 与 `REPORT_ASSISTANT_LLM_ENABLED=true`。密钥只能放入本地 `.env`，不得写入代码、测试、文档或日志。

核心数据库表包括 `report_generation`、`report_schedule`、`report_action`、`application_material_item`、`crm_lead_status_history`、`marketing_channel_cost`、`customer_contract` 和 `customer_payment`。先执行 `20260711_01_sync_report_generation_contract.up.sql`，再按演示需要执行 Seed；Iteration 3 Seed 必须与 Cleanup 配对，不得清理正常业务数据。

团队合并时只登记以下公共入口，禁止整体覆盖：

- `routers/__init__.py`：注册 reports、assistant、schedule、action、data Router。
- `models/__init__.py`：导入报告 ORM，保证元数据完整。
- `main.py`：调用统一 Router 注册函数，不重复注册。
- `.env.example`：保留 `REPORT_LLM_*` 与 `REPORT_ASSISTANT_*` 占位配置。
- 前端 Router、Sidebar、API Catalog：保留报告中心、助手和数据维护入口。
