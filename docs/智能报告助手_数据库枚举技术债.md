# 智能报告助手数据库枚举技术债

## 当前契约

`REPORT_REGISTRY` 是报告类型的代码侧事实目录。`report_generation.report_type` 使用 `VARCHAR(64)`，因此新增 Registry 报告类型时，不需要同步扩展数据库 ENUM。

`report_generation.status` 和 `trigger_source` 仍使用 ENUM：

- `status`：`pending`、`generating`、`completed`、`failed`。
- `trigger_source`：`manual`、`schedule`、`retry`、`system`。

这两个字段代表有限状态机，新增状态或来源时必须通过版本化数据库迁移同步，不能只修改 Python 常量。

## 为什么本轮不把所有字段改为 VARCHAR

本轮目标是收口 Iteration 2 的数据库契约，而不是重新设计整个报告存储模型。保留有限状态 ENUM 可以让数据库拒绝未知状态，改动也与当前 ORM 和 `db_init.sql` 一致。

## 后续架构评审项

后续新增报告类型或任务状态前，应评审以下方案：

1. `report_type` 继续使用 VARCHAR，并通过应用层 Registry 白名单校验。
2. `status`、`trigger_source` 使用 VARCHAR + CHECK，降低 MySQL ENUM 迁移成本。
3. 继续使用 ENUM，但把“修改 Python 常量时必须同时新增迁移”纳入提交检查。

本轮采用方案 1 和方案 3，并通过 `tests/test_report_assistant_database_contract.py` 检查代码契约与迁移声明是否一致。
