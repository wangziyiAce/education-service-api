# 海外留学教育服务平台答辩材料设计规格

## 1. 交付目标

为海外留学教育服务平台制作一套 15 分钟以内的答辩材料，包括：

- 14 页 PowerPoint 答辩演示文稿；
- 与 PPT 页序严格对应的逐页答辩文本；
- 曾铁昭与龙梦婷两位主讲人的清晰分工、切换提示和演示提示；
- 仅使用代码、Git 记录、测试结果及项目交付文档能够支持的事实。

## 2. 受众与叙事目标

受众为项目课教师和同学。答辩不以罗列接口数量为主，而以“业务问题、技术链路、关键取舍、真实成果、问题复盘”为主线，证明团队具备需求拆解、FastAPI 后端开发、AI 能力接入、团队协作与项目交付能力。

核心叙事为：

> 团队围绕海外留学咨询的客户研判、客服咨询、企业运营、学生服务和管理报告五类场景，在三天开发加一天测试周期内完成统一平台；开发中根据联调成本主动调整架构，只让客服 Agent 保留 Dify + Python，其余模块使用纯 Python，从而提高可测试性和集成稳定性。

## 3. 页序与内容结构

1. 封面：项目名称、团队成员、双主讲人；
2. 项目背景：业务痛点与项目目标；
3. 五模块整体关系：从潜在客户到管理决策的业务闭环；
4. 系统全景：React、FastAPI、MySQL、LLM 和 Dify 的真实边界；
5. 客户研判：资料解析、规则匹配、画像和 CRM 转化；
6. 客服 Agent：课程、活动、报名、会话以及 Dify + Python 链路；
7. 企业智能助手：CRM、跟进、日报和组织协同；
8. 学生智能助手：请假、成绩、通知、投诉和多轮对话；
9. 智能报告：报告生成、周期比较、证据和管理行动；
10. 五模块如何串联：模块间的上游、下游与数据交接；
11. 完整请求流：用户输入到结果返回的工程链路；
12. 工程质量：权限、统一异常、幂等、数据来源、测试与安全；
13. 关键技术决策与 Vibe Coding：架构调整、合并覆盖、配置遗漏和契约漂移；
14. 项目成果、已知边界和总结。

五个业务模块均为答辩重点，每个模块独立占一页，并统一回答业务问题、目标用户、核心能力、输入/处理/输出、代码位置、上下游关系和真实实现边界。

## 4. 双主讲分工

### 曾铁昭

- 主讲第 1—3、6—7、12 页；
- 负责开场、业务目标、五模块总关系、客服 Agent、企业助手和工程质量；
- 在跨模块关系页讲清客户进入、客服承接与企业协同部分；
- 重点讲解客服 Agent 的 Dify + Python 调用链。

### 龙梦婷

- 主讲第 4—5、8—9、13—14 页；
- 负责总体架构、客户研判、学生助手、智能报告、技术决策和项目总结；
- 在跨模块关系页讲清签约后学生服务、报告反馈与管理闭环；
- 重点回答 Vibe Coding 使用经验、踩坑和验证方法；
- 负责收尾并引导进入现场演示或提问环节。

### 共同讲解

- 第 10—11 页由两人接力完成；
- 曾铁昭讲客户资料进入、客服承接和企业协同；
- 龙梦婷讲学生服务、报告汇总、管理反馈和完整技术请求流。

总讲解时间目标约 13 分钟，预留 2 分钟处理停顿、切换和演示准备。

## 5. 视觉系统

视觉继承现有前端“国际教育编辑型门户”，不使用通用蓝色科技模板。

- 背景：羊皮纸白 `#F7F2EA`；
- 主要文字：深墨 `#15171C`；
- 主强调色：学院酒红 `#7A1F2B`；
- 次强调色：古铜 `#B68C5A`；
- 辅助线和浅底：石灰 `#D9D0C2`、卡片白 `#FFFDF8`；
- 字体：标题、正文、数据和注释统一使用思源黑体；
- 层级：只通过字号、字重、颜色和留白区分，不混用字体家族；
- 构图：16:9，编辑式网格、细线分隔、档案编号、不对称图文；
- 页面：避免密集卡片墙、彩虹渐变、玻璃拟态和大段代码截图；
- 图片：优先使用项目现有前端截图和界面局部，不使用无关素材图。

建议字号：封面标题 50—56 pt，页标题 35—40 pt，副标题 24—28 pt，正文 17—21 pt，注释不低于 14 pt。

## 6. 内容真实性边界

- 客服 Agent：明确为 Dify + Python；
- 客户研判、企业助手、学生助手、智能报告：明确为纯 Python；
- 不把计划中的 RAG、Redis、Celery、多 Agent、Docker 或生产部署写成已完成；
- 不把 Mock 或未验证的真实环境链路写成已上线；
- 测试数据在制作时以最新实际命令结果为准；
- PPT 使用“500+ 后端自动化测试”等稳定表述时，讲稿中补充当前实际结果及未关闭项；
- Dify 在线发布、真实外部 LLM 和真实数据库状态若缺少证据，使用“代码侧已接入，尚未完成现场环境验收”。

## 7. 答辩文本格式

答辩文本按 PPT 页序编写，每页固定包含：

- 页码与页面标题；
- 主讲人；
- 建议用时；
- 可直接朗读的讲稿；
- 切页或主讲交接提示；
- 需要现场操作时的演示提示；
- 可能被追问的关键点仅在相关页面补充。

讲稿使用自然口语，不照读 PPT，不虚构个人贡献，不使用“完全生产可用”等过度承诺。

## 8. 验证标准

- PPT 共 14 页且页序与讲稿完全一致；
- 两位主讲人的分工、交接和时间合计明确；
- 每页 PPT 均完成渲染并逐页检查；
- 不存在文字溢出、遮挡、标题换行或字体替换；
- 前端风格、色彩和思源黑体在所有页面保持一致；
- 技术栈边界与项目 AGENTS.md 一致；
- 测试数据、模块数量、路由和数据库事实均有仓库证据；
- 必问的 Vibe Coding 问题在第 11 页有直接回答。
