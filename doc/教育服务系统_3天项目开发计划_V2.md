| 项目 | 内容 |

| 分配模式 | **按模块分配**，每人负责一个完整模块（接口 + Dify + 文档） |

> **V2.0 核心变化**：从 V1.0 的"按角色分配"升级为"按模块分配"。5 人各负责一个完整业务模块（含后端接口、ORM、Dify Chatflow、种子数据、模块文档），确保工作量均衡，每人都有独立可演示成果。

---

# 一、项目开发目标

本计划用于指导 5 人小组在 3 天内完成教育服务系统 MVP 版本。

3 天内必须完成以下目标：

1. FastAPI 后端项目可启动。
2. MySQL 数据库可连接。
3. 核心业务接口可通过 Swagger 测试。
4. Dify Chatflow 能调用 FastAPI 接口。
5. **5 个模块各形成可演示闭环**：客服 Agent、企业助手、学生助手、智能报告、前端展示。
6. 前端或 Dify WebApp 可展示核心功能。
7. 完成答辩所需文档、截图、PPT 素材和演示脚本。
8. **每个成员都有独立的模块成果可用于答辩展示。**

## 2.3 工作量平衡分析

```
成员 A: 基础设施(半天) → 每日协调(持续) → 答辩统筹(Day3)
        接口数:1   表:1   Dify节点:0   前端:0   文档:5

成员 B: 课程活动接口(Day1) → 活动报名+会话(Day1下午) → Dify联调(Day1晚-Day2)
        接口数:5   表:5   Dify节点:4   前端:0   文档:2
        模块闭环: 用户问课程 → Dify调接口 → 返回推荐

成员 C: CRM接口(Day1) → 日报接口(Day2上午) → Dify联调(Day2)
        接口数:7   表:3   Dify节点:3   前端:0   文档:2
        模块闭环: 员工录入客户 → 查客户 → 更新状态 → 写日报

成员 D: 请假接口(Day1) → 心理+投诉(Day2上午) → Dify联调(Day2)
        接口数:8   表:5   Dify节点:4   前端:0   文档:2
        模块闭环: 学生请假 → 老师审批 → 状态更新 → 通知

成员 E: 报告接口(Day1) → 前端页面(Day1下午-Day2) → 答辩材料(Day3)
        接口数:3   表:2   Dify节点:2   前端:5   文档:3
        模块闭环: 触发报告 → 聚合数据 → Dify生成 → 前端展示
```

> **结论**：B/C/D 工作量相当（接口数 5-8 个，表数 3-5 张，Dify 节点 3-4 个，约 2 天纯开发 + 联调）。A 基础设施集中在前半天，后续以协调为主。E 跨后端、前端、答辩，时间线最长但密度适中。

# 十六、结论

3 天开发周期下，本项目要以"可演示主链路"为核心，不追求生产级完整度。

核心策略：

1. **按模块分配**：5 人各负责一个完整模块，有独立可交付成果，答辩时每人都有展示内容。
2. **并行推进**：B/C/D 三个模块完全并行开发，互不等待，最大化 Day1-Day2 的开发效率。
3. **先闭环再完善**：每个模块先跑通最简链路（接口→Dify→回复），再加功能和优化。
4. **数据驱动**：先用种子数据验证接口正确性，再接入 AI。

最终目标不是功能堆砌，而是证明以下能力已经跑通：

```text
Dify 能理解用户意图；
FastAPI 能提供业务接口；
MySQL 能保存业务数据；
Dify 能调用 FastAPI；
前端或 Dify WebApp 能展示结果；
项目具备 AI 应用落地闭环；
每个成员都有独立的模块成果。
```

只要这条链路稳定，答辩时就能清晰说明项目架构、技术选型、业务价值和后续扩展方向。
