## 3. 核心定稿结论

### 3.1 架构主线

系统继续采用“Dify 做大脑，FastAPI 做手脚，MySQL 做记忆，前端做操作面板”的职责划分。

- Dify 负责意图识别、RAG、对话生成和工作流编排。
- FastAPI 负责接口、权限、业务规则、事务和数据读写。
- MySQL 是业务主数据唯一真实数据源。
- Vue 前端不直接操作数据库，所有数据来自 FastAPI。

### 3.2 MVP 范围

3 天 MVP 不追求一次性完成所有页面和生产级组件。最终口径为：

| 层面 | P0 必须跑通 | P1/P2 增强 |
|---|---|---|
| 演示入口 | Dify WebApp + Swagger | Vue 后台完整页面 |
| AI 能力 | 客服 Agent、报告生成 | 客户研判、心理预警、多 Agent |
| 后端接口 | 认证、课程、活动报名、CRM、学生服务、报告 | 管理后台、扩展查询、更多异步任务 |
| 数据库 | P0 主链路表 | 知识库、心理、客户研判、通知、定时任务等扩展表 |
| 工程组件 | FastAPI + MySQL + Dify | Redis、Celery、MinIO、Milvus、Alembic |

### 3.3 客户研判口径

客户研判属于业务核心模块，但考虑 3 天 MVP 风险，本次定稿明确：

- 文本资料研判链路可作为 P1 验证。
- PDF / Excel 上传解析作为后续增强。
- 前端客户研判上传页不阻塞 P0 主链路验收。

### 3.4 前端口径

前端 V1.1 将 Vue 后台从“P0 必做页面”调整为“P1 增强页面”。

- P0：Dify WebApp 展示 AI 对话，Swagger 展示接口能力。
- P1：登录、CRM、学生服务、报告详情等 Vue 页面按时间完成。
- P2：客户研判上传、心理预警、看板等复杂页面。

## 4. 关键一致性校验

| 校验项 | 结论 |
|---|---|
| API 前缀 | 统一为 `/api/v1` |
| 数据库访问 | MVP 统一同步 SQLAlchemy + PyMySQL |
| 外键策略 | 数据库禁用物理外键，应用层维护一致性 |
| 字段命名 | 数据库、API、前端类型统一使用 `snake_case` 字段名 |
| Dify 鉴权 | Dify 调 FastAPI 使用 `DIFY_SERVICE_TOKEN` |
| Dify 白名单 | 仅课程、活动、报名、少量 P1 查询/写入接口开放 |
| 事务边界 | 事务中不调用 Dify、邮件、短信等外部服务 |
| 异步任务 | 报告生成、客户研判使用 BackgroundTasks + 独立 Session |
| 敏感数据 | Token、API Key、心理原文等不进入普通日志 |

## 5. 定稿建议

1. 开发阶段以新版定稿文档为准，旧版本只作为历史参考。
2. `db_init.sql` 应按数据库 V2.1 再做一次字段、索引、无外键检查。
3. FastAPI 路由和 Pydantic Schema 应按 API V1.2 冻结 P0 接口。
4. Dify HTTP 节点优先配置课程查询、活动查询、活动报名三条 P0 链路。
5. Vue 后台按前端 V1.1 的 P1 顺序开发，不作为 P0 演示阻塞项。
6. 答辩演示优先准备 Dify WebApp、Swagger、数据库种子数据和接口截图/录屏兜底。
