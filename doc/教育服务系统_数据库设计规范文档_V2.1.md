## 1. 设计总则

### 1.1 数据库选型

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 数据库 | MySQL 8.0 | 团队熟悉，与 db_init.sql 兼容 |
| 字符集 | `utf8mb4` | 支持 emoji 和全字符集 |
| 排序规则 | `utf8mb4_unicode_ci` | 大小写不敏感，广泛兼容 |
| 存储引擎 | InnoDB | 支持事务、行级锁、MVCC |
| 访问方式 | SQLAlchemy 同步 + PyMySQL | MVP 降低异步排错成本 |
| 连接串 | `mysql+pymysql://root:password@localhost:3306/education_service?charset=utf8mb4` | - |
| 连接池 | SQLAlchemy QueuePool, pool_size=10, max_overflow=20 | 生产环境建议值 |

### 1.2 设计原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | **唯一真实数据源** | MySQL 是所有业务数据的唯一来源，Dify 不存储业务数据 |
| 2 | **表名/字段名小写下划线** | 统一命名风格，避免大小写问题 |
| 3 | **主键统一为 `id`** | 使用 `BIGINT UNSIGNED AUTO_INCREMENT` |
| 4 | **时间字段统一** | `create_time` + `update_time`，使用 `DATETIME` 类型 |
| 5 | **状态字段用英文枚举** | 禁止中文自由文本做状态，统一 `ENUM` 或 `VARCHAR` |
| 6 | **文件只存路径** | 文件不存入数据库，只保存 `file_url` 路径 |
| 7 | **AI 输出双存储** | 保存原始文本 + 结构化 JSON |
| 8 | **关键写入记录操作人** | 涉及数据变更的表需记录 `operator_id` 或 `xxx_by` |
| 9 | **禁用物理外键** | ⭐ 所有表之间仅逻辑关联，通过索引 + 应用层维护关系 |
| 10 | **每表必有 COMMENT** | 表和字段均添加中文注释 |
| 11 | **软删除优先** | 核心数据不物理删除，使用 `is_deleted` 或 `status` 标记 |
| 12 | **读写分离预留** | 关联字段命名统一 `{entity}_id`，为后续读写分离留接口 |

### 1.3 数据库命名

| 项目 | 值 |
|------|-----|
| 数据库名 | `education_service` |
| 默认字符集 | `utf8mb4` |
| 默认排序规则 | `utf8mb4_unicode_ci` |

### 1.4 全链路评审定稿结论（V2.1）

数据库 V2.1 延续 V2.0 的无物理外键策略，不新增破坏性 DDL。经与需求规格、总体架构 V1.2、API V1.2、Dify V1.1、前端 V1.1 交叉评审后，定稿结论如下：

| 评审项 | 定稿结论 |
|---|---|
| 数据源边界 | MySQL 仍为业务主数据唯一真实数据源，Dify 不直接读写数据库 |
| 外键策略 | 继续禁用物理外键，通过 `{entity}_id`、索引、Service 校验和一致性脚本维护关系 |
| MVP 建表范围 | P0 表支撑认证、课程活动、报名、CRM、学生服务、报告生成主链路；P1 表支撑客户研判、心理预警、知识库和扩展演示 |
| API 字段对齐 | API V1.2 继续使用数据库字段名作为 JSON 字段名，前端不做二次字段翻译 |
| 事务边界 | 活动报名、状态变更、审批等写操作必须在 FastAPI 内完成，事务中不调用 Dify、邮件、短信等外部服务 |
| 定稿状态 | 本文档可作为 `db_init.sql` 校准和 ORM Model 编写依据 |

## 3. 字段类型规范

### 3.1 主键与 ID 字段

| 场景 | 类型 | 说明 |
|------|------|------|
| 主键 | `BIGINT UNSIGNED NOT NULL AUTO_INCREMENT` | 统一使用，支持大数据量 |
| 外键关联 | `BIGINT UNSIGNED DEFAULT NULL` | 与主键类型一致，**仅逻辑关联** |
| 用户ID关联 | `BIGINT UNSIGNED` | 关联 `sys_user.id` |

### 3.2 字符串字段

| 场景 | 类型 | 说明 |
|------|------|------|
| 短文本（≤64字符） | `VARCHAR(64)` | 姓名、编码、标签 |
| 中等文本（≤255字符） | `VARCHAR(255)` | 标题、文件名、URL路径 |
| 长URL | `VARCHAR(512)` | 文件URL、头像URL |
| 长文本 | `TEXT` | 描述、原因、备注 |
| 超长文本 | `MEDIUMTEXT` | 报告HTML内容 |
| 固定长度编码 | `VARCHAR(32)` | 学号、工号、会话ID |

### 3.3 数字字段

| 场景 | 类型 | 说明 |
|------|------|------|
| 金额 | `DECIMAL(10,2)` | 价格，精确到分 |
| 评分/百分比 | `DECIMAL(5,2)` | 匹配分数(0-100)、成绩 |
| 小数（学分） | `DECIMAL(3,1)` | 学分 |
| 整数计数 | `INT` | 报名人数、Token消耗 |
| 排序权重 | `INT DEFAULT 0` | sort_order |
| 布尔标记 | `TINYINT DEFAULT 0` | 0=否, 1=是 |
| 状态标记 | `TINYINT DEFAULT 1` | 1=启用, 0=禁用 |

### 3.4 时间字段

| 场景 | 类型 | 说明 |
|------|------|------|
| 创建/更新时间 | `DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP` | 自动记录 |
| 更新时间 | `DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` | 自动更新 |
| 业务日期 | `DATE` | 日报日期、记录日期 |
| 可选时间 | `DATETIME DEFAULT NULL` | 审批时间、关闭时间 |

### 3.5 JSON 字段

| 场景 | 类型 | 说明 |
|------|------|------|
| 结构化配置 | `JSON` | 规则配置、标签、进度 |
| AI 输出结果 | `JSON` | 研判结果、报告内容 |
| 数组数据 | `JSON` | 推荐项目、接收人列表 |

> **JSON 字段查询优化**: 高频查询的 JSON 内部字段，考虑使用 MySQL 8.0 的 `JSON_EXTRACT` 建立**生成列 + 索引**：
> ```sql
> ALTER TABLE report_generation 
>   ADD COLUMN report_type_virtual VARCHAR(64) 
>   GENERATED ALWAYS AS (JSON_UNQUOTE(JSON_EXTRACT(report_content, '$.type'))) STORED;
> CREATE INDEX idx_report_type_virtual ON report_generation(report_type_virtual);
> ```

### 3.6 枚举字段

MVP 阶段推荐使用 `ENUM` 类型（可读性好，数据库层面约束），后续可升级为字典表。

```sql
-- 推荐写法
`status` ENUM('pending','approved','rejected','cancelled') NOT NULL DEFAULT 'pending' COMMENT '状态'

-- 不推荐
`status` VARCHAR(32) COMMENT '状态'  -- 缺少数据库约束
```

> **ENUM 扩展注意事项**: `ENUM` 类型使用 `ALTER TABLE ... MODIFY COLUMN` 添加新值时会锁表（MySQL 8.0.22+ 部分改善）。如果预期枚举值频繁变化，建议使用 `VARCHAR(32)` + 应用层校验。

## 4. 索引设计规范

### 4.1 索引原则

| 原则 | 说明 |
|------|------|
| **主键必建** | 每表必须有自增主键 |
| **唯一约束加唯一索引** | 用户名、会话ID、联合唯一 |
| **高频查询加普通索引** | status、逻辑外键ID、日期字段 |
| **逻辑外键字段必加索引** | 所有 `xxx_id` 字段（替代物理外键的索引需求） |
| **复合索引按查询频率** | 最常用的查询条件放前面 |
| **避免过度索引** | 写入频繁的表索引不超过 5-6 个 |
| **全文索引按需建** | 知识库、FAQ 等内容检索场景 |
| **覆盖索引优化** | 高频查询尽量使用覆盖索引避免回表 |
| **前缀索引** | 超长 VARCHAR 字段（如 URL），使用前缀索引减少空间 |

### 4.2 索引设计检查清单

```text
[ ] 主键: id (BIGINT UNSIGNED AUTO_INCREMENT)
[ ] 唯一约束字段: UNIQUE KEY
[ ] 逻辑外键字段: KEY idx_{field}  ← 禁用物理外键后更重要
[ ] 状态查询字段: KEY idx_status
[ ] 时间范围查询: KEY idx_{time_field}
[ ] 用户隔离查询: KEY idx_{user_field}
[ ] 排序字段: KEY idx_{sort_field}（如 ORDER BY create_time DESC 高频）
[ ] 全文检索字段: FULLTEXT KEY (仅知识库类表)
[ ] 覆盖索引: 是否能用覆盖索引避免回表？
```

### 4.3 典型索引示例

```sql
-- 单列索引（最常用）
KEY `idx_status` (`status`)
KEY `idx_student_id` (`student_id`)

-- 唯一索引
UNIQUE KEY `uk_username` (`username`)
UNIQUE KEY `uk_session_id` (`session_id`)

-- 复合唯一索引（防重复）
UNIQUE KEY `uk_employee_date` (`employee_id`, `report_date`)

-- 复合索引（联合查询优化）
KEY `idx_student_date` (`student_id`, `record_date`)

-- 覆盖索引（避免回表，查询只需索引列）
KEY `idx_cover_status_time` (`status`, `create_time`, `id`)

-- 前缀索引（超长 VARCHAR）
KEY `idx_file_url` (`file_url`(100))

-- 全文索引（知识库检索，ngram 支持中文分词）
FULLTEXT KEY `ft_content` (`title`, `content`) WITH PARSER ngram
```

### 4.4 索引性能监控

```sql

-- 查看冗余索引
SELECT * FROM sys.schema_redundant_indexes;

-- 查看全表扫描的查询

```

## 5. 外键策略：全面禁用物理外键

### 5.1 为什么禁用物理外键

| 问题 | 物理外键的表现 | 无外键的优势 |
|------|--------------|------------|
| **行锁升级** | 子表 INSERT/UPDATE 时父表相关行被加共享锁 | 无额外锁，并发性能高 |
| **级联操作不可控** | `ON DELETE CASCADE` 可能意外删除大量数据 | 应用层显式控制删除逻辑 |
| **分库分表困难** | 物理外键无法跨库 | 逻辑关联天然支持水平拆分 |
| **数据迁移/归档** | 外键约束阻塞批量操作 | 批量操作不受限 |
| **死锁风险** | 多表关联操作易触发死锁 | 死锁概率大幅降低 |
| **测试数据插入** | 必须按依赖顺序插入 | 可任意顺序插入，便于单测 |

### 5.2 逻辑外键替代方案

#### 规则 1：逻辑外键字段命名统一

```text
所有关联字段必须以被关联表的表名 + _id 命名：
  sys_user.id  →  user_id, owner_employee_id, assignee_id, approver_id
  crm_lead.id  →  lead_id
  event_lecture.id → event_id
```

#### 规则 2：逻辑外键字段必须建立普通索引

```sql
-- 替代 FOREIGN KEY 的索引要求
KEY `idx_student_id` (`student_id`)   -- 原 FOREIGN KEY (student_id) REFERENCES sys_user(id)
KEY `idx_event_id` (`event_id`)       -- 原 FOREIGN KEY (event_id) REFERENCES event_lecture(id)
```

#### 规则 3：应用层维护数据一致性

```python
# 插入子表前，校验父表记录存在
def create_registration(event_id: int, user_id: int):
    # 1. 校验活动存在
    event = db.query(EventLecture).filter_by(id=event_id).first()
    if not event:
        raise NotFoundError("活动不存在")
    
    # 2. 校验用户存在
    user = db.query(SysUser).filter_by(id=user_id).first()
    if not user:
        raise NotFoundError("用户不存在")
    
    # 3. 执行业务操作
    registration = EventRegistration(event_id=event_id, user_id=user_id)
    db.add(registration)
    db.commit()
```

#### 规则 4：删除父表记录时，应用层处理关联数据

```python
# 删除活动前，处理关联的报名记录
def delete_event(event_id: int):
    # 1. 将关联报名记录标记为已取消（软处理）
    db.query(EventRegistration).filter_by(event_id=event_id).update(
        {"status": "cancelled"}
    )
    # 2. 软删除活动（而非物理删除）
    db.query(EventLecture).filter_by(id=event_id).update(
        {"status": "cancelled", "is_deleted": 1}
    )
    db.commit()
```

### 5.3 数据一致性校验脚本

建议在低峰期运行一致性校验：

```sql
-- 检查孤立的报名记录（event 已删除但报名记录未处理）
SELECT er.id, er.event_id 
FROM event_registration er 
LEFT JOIN event_lecture el ON er.event_id = el.id 
WHERE el.id IS NULL AND er.status != 'cancelled';

-- 检查孤立的跟进记录
SELECT cf.id, cf.lead_id 
FROM crm_follow_up cf 
LEFT JOIN crm_lead cl ON cf.lead_id = cl.id 
WHERE cl.id IS NULL;

-- 检查学生信息与用户表的 1:1 关系
SELECT si.id, si.user_id 
FROM student_info si 
LEFT JOIN sys_user su ON si.user_id = su.id 
WHERE su.id IS NULL OR su.user_type != 'student';
```

### 5.4 外键策略总结表

| 策略 | 实现方式 |
|------|---------|
| 关系表达 | 字段命名 `{entity}_id` + 索引 |
| 存在性校验 | 应用层插入前 SELECT 校验 |
| 级联删除 | 应用层显式处理（标记状态或物理删除） |
| 数据一致性 | 定时校验脚本 + 告警 |
| 开发规范 | Code Review 检查：所有 `xxx_id` 字段是否有对应索引 |

**并发控制（无外键版本）**:

```python
# 应用层并发控制：使用 SELECT ... FOR UPDATE 行级锁
def register_for_event(event_id: int, user_id: int):
    with db.begin():
        # 1. 悲观锁锁定活动行
        event = db.query(EventLecture).filter_by(
            id=event_id, status='upcoming'
        ).with_for_update().first()
        
        if not event:
            raise BusinessError("活动不存在或已结束")
        if event.current_participants >= (event.max_participants or 999999):
            raise BusinessError("名额已满")
        
        # 2. 校验不重复报名（uk_event_user 唯一索引兜底）
        existing = db.query(EventRegistration).filter_by(
            event_id=event_id, user_id=user_id
        ).first()
        if existing:
            raise BusinessError("已报名，请勿重复报名")
        
        # 3. 插入报名记录
        reg = EventRegistration(event_id=event_id, user_id=user_id)
        db.add(reg)
        
        # 4. 原子更新名额计数
        db.execute(
            update(EventLecture)
            .where(EventLecture.id == event_id)
            .values(current_participants=EventLecture.current_participants + 1)
        )
    # 事务自动提交
```

### 12.3 禁止事项

| 禁止 | 原因 | 替代方案 |
|------|------|---------|
| `SELECT *` | 明确指定需要的字段 | 列出具体字段 |
| 拼接 SQL 字符串 | 防 SQL 注入 | 参数化查询 |
| 不带 LIMIT 的查询 | 防止全表扫描 | 必须加 LIMIT |
| 在循环中执行 SQL | 性能极差 | 批量操作 |
| 中文做状态值 | 索引效率低，可读性差 | 英文枚举 |
| 物理删除核心数据 | 无法追溯 | 软删除或状态标记 |
| 密码明文存储 | 安全风险 | bcrypt 哈希 |
| 文件二进制存入数据库 | 数据库膨胀 | 只存路径 URL |
| **使用物理外键** | ⭐ 行锁升级、分库分表困难 | 逻辑关联 + 应用层维护 |
| `LIKE '%keyword%'` 前缀通配 | 索引失效，全表扫描 | 全文索引或 Elasticsearch |
| 大事务 | 锁持有时间长 | 拆分小事务 |

### 14.1 应用层校验规范

# ============================================
# 数据一致性校验装饰器（utils/validators.py）
# ============================================

from functools import wraps
from sqlalchemy.orm import Session

def validate_entity_exists(entity_model, field_name: str, error_msg: str):
    """校验逻辑外键指向的实体是否存在"""
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            entity_id = kwargs.get(field_name)
            if entity_id is not None:
                session: Session = kwargs.get('db')
                exists = session.query(
                    session.query(entity_model)
                    .filter_by(id=entity_id)
                    .exists()
                ).scalar()
                if not exists:
                    raise ValueError(error_msg.format(id=entity_id))
            return func(*args, **kwargs)
        return wrapper
    return decorator

# 使用示例

### 14.2 级联操作规范

```python
# ============================================
# 级联删除/更新示例（services/cascade.py）
# ============================================

def soft_delete_lead(db: Session, lead_id: int):
    """软删除意向客户，同步处理关联数据"""
    with db.begin():
        # 1. 标记关联跟进记录为无效
        db.query(CRMFollowUp).filter_by(lead_id=lead_id).update(
            {"is_deleted": 1}, synchronize_session=False
        )
        # 2. 软删除客户
        result = db.query(CRMLead).filter_by(id=lead_id).update(
            {"status": "lost", "lost_reason": "数据清理", "is_deleted": 1}
        )
        if result == 0:
            raise NotFoundError("客户不存在")
    return {"lead_id": lead_id, "status": "deleted"}

def cancel_event(db: Session, event_id: int):
    """取消活动，批量取消报名"""
    with db.begin():
        # 1. 锁定活动行
        event = db.query(EventLecture).filter_by(
            id=event_id, status='upcoming'
        ).with_for_update().first()
        if not event:
            raise BusinessError("活动不存在或已结束")
        
        # 2. 批量取消报名
        db.query(EventRegistration).filter_by(
            event_id=event_id, status='registered'
        ).update({"status": "cancelled"}, synchronize_session=False)
        
        # 3. 更新活动状态
        event.status = 'cancelled'
```

### 14.4 事务边界规范

```text
事务原则:
  [ ] 一个业务操作 = 一个事务（原子性）
  [ ] 事务尽量短小（< 1 秒）
  [ ] 不在事务中调用外部 API（Dify、邮件、短信）
  [ ] 不在事务中进行文件 IO
  [ ] 使用 with db.begin() 确保事务正确提交/回滚
  [ ] 高并发场景使用 SELECT ... FOR UPDATE（悲观锁）
  [ ] 低并发场景可使用乐观锁（version 字段）
```

### 15.2 变更管理原则

| 原则 | 说明 |
|------|------|
| 新增表 | 更新 `db_init.sql`，添加 `DROP TABLE IF EXISTS` + `CREATE TABLE` |
| 新增字段 | 在 `db_init.sql` 对应表中添加 `ALTER TABLE ... ADD COLUMN` 语句 |
| 修改字段 | 谨慎操作，优先新增字段而非修改已有字段 |
| 删除字段 | MVP 阶段尽量避免，如需删除需团队确认 |
| 变更记录 | 在 `db_init.sql` 顶部注释中记录变更历史 |
| 外键检查 | 每次变更后确认无物理外键残留 |
