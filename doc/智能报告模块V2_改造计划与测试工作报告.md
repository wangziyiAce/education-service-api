## 3. 改造背景与业务目标

海外留学咨询平台中，智能报告模块不是一个单纯的“让大模型写总结”的功能，而是管理层用来做业务分析、风险预警和运营复盘的后端分析模块。

旧版 MVP 的问题主要有三个：

1. 业务数字和解释文本耦合在一起，容易让模型“看起来说得对”，但数字来源不可追溯。
2. 报告类型扩展依赖单文件堆逻辑，后续新增报告会越来越难维护。
3. 缺少定时调度、失败重试、行动闭环、数据质量说明和接口契约测试。

V2 的目标是把报告模块从“文本生成工具”升级为“管理分析系统”：

- 数据来自数据库和规则引擎；
- 任务由统一编排层管理；
- 每类报告有独立聚合器和 Schema；
- 报告能定时生成、失败可重试；
- 报告建议能转行动项，形成管理闭环；
- 前端可以根据 `report_type + schema_version` 选择对应渲染方式。

## 4. 架构改造内容

### 4.0 V1 与 V2 的关键差异

| 维度 | V1 / MVP 思路 | V2 改造结果 |
|---|---|---|
| 报告结构 | 多类报告复用通用字段 | 每类报告拥有独立 Schema，前端按 `report_type + schema_version` 渲染 |
| 类型扩展 | 容易堆叠 `if report_type == ...` | 通过注册表登记聚合器、模板、权限、周期和 Schema |
| 指标来源 | 容易依赖 AI 直接生成文本和数字 | 指标由 SQL 和规则引擎计算，AI 只解释 |
| 任务生命周期 | 生成记录较简单 | 支持 `pending / generating / completed / failed` 和失败重试 |
| 定时能力 | 主要依赖手动接口触发 | 支持独立 APScheduler 调度与周期幂等 |
| 前端对接 | 通用结构展示，差异化不足 | `/reports/types` 返回类型、权限、模板和可用过滤条件 |
| 管理闭环 | 报告建议停留在文本 | 建立 `report_action`，建议可转为行动并跟踪结果 |
| 隐私边界 | 敏感数据处理边界不够明确 | 心理预警等只进入聚合统计，不保存或输出学生原文 |

### 4.1 服务层拆分

本次将原先集中在 `services/report_service.py` 的逻辑，升级为新的 `services/reporting/` 分层包：

| 文件 | 作用 |
|---|---|
| `services/reporting/orchestrator.py` | 报告任务编排：创建任务、生成报告、重试、保存状态 |
| `services/reporting/registry.py` | 报告类型注册表：登记报告类型、聚合器、Schema、权限、模板 |
| `services/reporting/aggregators.py` | 数据聚合器：每类报告独立查询数据库并计算指标 |
| `services/reporting/rules.py` | 确定性规则：风险评分、ROI、SLA、转化率等 |
| `services/reporting/ai_generator.py` | AI 解释层：对聚合结果生成解释，并校验输出结构 |
| `services/reporting/renderer.py` | HTML 渲染层：按报告类型使用后端模板生成 HTML |
| `services/reporting/schemas.py` | 报告内容结构：定义各类报告的 Pydantic 内容模型 |

这种拆分解决了一个核心工程问题：新增报告时不再到处写 `if report_type == ...`，而是注册一个类型、写一个聚合器、配一个 Schema 和模板即可。

### 4.2 报告生成链路

当前报告生成链路可以概括为：

```text
前端请求生成报告
→ FastAPI 接口校验 JWT 和请求体
→ orchestrator 创建 report_generation 任务
→ BackgroundTasks 或 APScheduler 触发生成
→ registry 找到报告定义
→ aggregator 查询数据库并计算指标
→ rules 计算风险/ROI/SLA/转化率
→ ai_generator 生成解释并校验 Schema
→ renderer 使用模板生成 HTML
→ orchestrator 保存报告内容、HTML、快照和状态
→ 前端查询报告详情
```

这里最重要的设计点是：AI 不直接生成业务数字。数字先由数据库和规则引擎产出，AI 只解释“这些数字代表什么、风险在哪里、下一步建议是什么”。

### 4.3 AI / Dify 边界控制

V2 保留 Dify Chatflow 对报告解释能力的支持，但明确限制 AI 的职责边界：

- AI 可以补充 `summary`、`explanation`、管理建议等解释性内容；
- AI 不允许改写风险分、转化率、CPL、CAC、ROI、SLA 超时率、行动完成率等业务指标；
- 后端会把 `aggregated_data`、`expected_schema` 和 `data_quality` 作为 Chatflow 的 `inputs` 传给 AI；
- Chatflow 的 `query` 只负责触发“基于这些聚合数据生成解释”的对话任务；
- Chatflow 返回结果通常在 `answer` 中，后端需要解析结构化内容后再校验 Schema；
- AI 输出必须通过 Pydantic Schema 校验；
- 第一次校验失败时允许携带错误信息修复一次；
- 第二次仍失败则报告任务进入 `failed`；
- 未启用 Dify 时使用本地解释模式，并在数据质量中明确标记，避免把开发模式误认为正式 AI 结果。

这个设计能体现 AI 应用开发中的一个关键工程意识：大模型适合做解释、表达和建议，但核心业务数字必须来自可追溯的数据计算。

## 5. 支持的 10 类报告

本次 V2 支持 10 类报告：

| 类型编码 | 报告名称 | 业务价值 |
|---|---|---|
| `customer_ops` | 客户经营分析报告 | 观察客户转化、阶段停留、长期未跟进和流失风险 |
| `daily_summary` | 员工日报汇总报告 | 汇总员工日常提交、进展、风险和下一步计划 |

| `sales_funnel` | 销售漏斗报告 | 分析线索阶段、转化率、停留时间和顾问跟进表现 |

| `service_sla` | 服务 SLA 报告 | 覆盖投诉、行政服务、心理预警的响应和解决时效 |

新增 5 类高价值报告分别是：

- `application_risk`
- `sales_funnel`

## 6. 数据库建设内容

### 6.1 核心任务表

#### `report_generation`

一条记录代表一次报告生成尝试，而不是永远覆盖同一份报告。

它的核心作用是：

- 保存报告类型、标题、周期、状态；
- 保存 `schema_version`；
- 保存触发来源 `trigger_source`；
- 保存请求过滤条件 `request_filters`；
- 保存聚合数据快照；
- 记录失败原因、错误码、重试次数；
- 记录开始时间和完成时间；
- 支持失败后创建新的重试记录，而不是覆盖旧失败记录。

#### `report_schedule`

用于定时报告计划。

它的核心作用是：

- 保存 cron 表达式；
- 保存启停状态；
- 保存时区；
- 保存周期规则，如 `previous_day`、`previous_week`、`previous_month`；
- 保存标题模板和过滤条件；
- 记录最近运行状态、错误信息和下次运行时间；
- 通过计划 ID 和统计周期形成幂等键，避免同一周期重复生成。

### 6.2 新增事实数据表

| 表名 | 作用 |
|---|---|
| `application_material_item` | 保存申请材料、必填状态、截止日期、提交状态，用于申请风险报告 |
| `crm_lead_status_history` | 保存客户阶段变化历史，用于销售漏斗和阶段停留分析 |
| `marketing_channel_cost` | 保存渠道投放成本，用于渠道 ROI 计算 |
| `customer_contract` | 保存客户合同金额、签约时间和状态 |
| `customer_payment` | 保存实际回款流水，用于 ROI 的真实回款计算 |
| `report_action` | 保存报告建议转行动后的执行闭环 |
| `student_feedback_ticket` | 保存投诉/反馈工单及响应、解决时间 |
| `student_psych_alert` | 保存心理预警等级、状态和首次跟进时间 |

这些事实表的价值在于：报告不是靠模型“总结感觉”，而是依赖可追溯的数据源。

## 8. 接口兼容性修复

在接口冒烟测试中，发现了一类真实前后端对接风险：前端计划字段名和后端内部字段名不完全一致。

本次在 `schemas/report.py` 中通过 Pydantic `AliasChoices` 做了兼容，服务层仍使用统一内部字段。

| 前端常用字段 | 后端内部字段 | 说明 |
|---|---|---|
| `title` | `report_title` | 报告生成标题 |
| `suggestion` | `suggestion_text` | 报告建议转行动项 |
| `is_required` | `required` | 申请材料是否必填 |
| `spend_date` | `cost_date` | 渠道成本日期 |
| `cost` | `cost_amount` | 渠道成本金额 |
| `paid_time` | `payment_time` | 回款时间 |
| `reason` | `change_reason` | CRM 阶段变更原因 |

这类修复虽然代码量不大，但非常接近真实企业项目中的接口联调问题：后端字段设计、前端表单命名、交付文档命名必须尽量兼容，否则接口“代码没错”，但联调会失败。
