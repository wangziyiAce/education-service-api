# 一、指导书目标

本指导书用于指导 5 人小组在 3 天内完成教育服务系统的最小可演示版本开发。

本项目不是完整生产级系统，而是面向课程项目、答辩演示和 AI 应用落地验证的 MVP 项目。开发过程中必须优先保证：

1. 主业务链路可跑通。
2. 核心接口可调用。
3. Dify 能成功调用 FastAPI。
4. 数据能正常入库和查询。
5. 至少形成一条完整演示闭环。
6. 文档、代码、测试截图和答辩材料可交付。
7. **每个成员都有独立的、可展示的模块成果。**

工程推进原则：

```text
先闭环，再完善；
先接口，再页面；
先数据，再智能；
先演示，再优化；
先确定边界，再扩展功能；
按模块独立开发，并行推进不阻塞。
```

# 二、项目开发总流程

项目开发从需求规格说明书定稿开始，后续按以下流程推进：

```text
需求规格说明书定稿
        ↓
MVP 范围确认
        ↓
总体架构设计
        ↓
数据库设计（成员 A 主导，全员评审）
        ↓
API 接口设计（成员 A 统一规范，各模块负责人各自设计）
        ↓
Dify 工作流设计（各模块负责人各自设计）
        ↓
前端架构与页面设计（成员 E 主导）
        ↓
任务拆解与排期（按模块分配）
        ↓
┌─────────────────────────────────────────────────────┐
│  并行开发阶段（各模块负责人独立推进）                      │
│  A: 基础设施    B: 客服Agent   C: 企业助手              │
│  D: 学生助手    E: 报告+前端                           │
└─────────────────────────────────────────────────────┘
        ↓
Dify 联调（各模块负责人 + A 协调）
        ↓
前端页面接入（E 主导，各模块负责人配合）
        ↓
测试与问题修复（全员，各自负责模块）
        ↓
答辩材料整理（A + E 主导，全员贡献）
        ↓
项目交付
```

**非标设备类比**：

| 非标设备类比 | 软件项目对应内容 |

# 三、模块分配方案（核心）

## 3.1 分配原则

```
┌──────────────────────────────────────────────────────────────┐
│                      按模块分配原则                            │
│                                                              │
│  1. 每个成员负责一个完整的业务模块                               │
│  2. 模块内包含：后端接口 + ORM模型 + Dify工作流 + 种子数据       │
│  3. 模块间低耦合，减少等待依赖                                  │
│  4. 每个模块都有独立可演示的闭环                                 │
│  5. 工作量通过接口数量、表数量、Dify节点数综合平衡               │
└──────────────────────────────────────────────────────────────┘
```

  │   被依赖: E (前端CRM页面)
  │
  ├── 学生助手 (D) ── 独立模块，无模块间依赖

# 五、总体架构设计流程

## 5.1 架构设计目标

总体架构设计要回答以下问题：

1. 系统有哪些模块？
2. 模块之间怎么通信？
3. Dify 和 FastAPI 谁负责什么？
4. MySQL、Redis、MinIO、Milvus 在项目中各自负责什么？
5. 发生异常时怎么处理？
6. 答辩时主链路如何展示？

## 5.2 推荐架构

```text
用户 / 前端页面 / Dify WebApp
        ↓
Dify Chatflow / Workflow
        ↓ HTTP 请求节点
FastAPI 后端服务
        ↓
MySQL 业务数据库
```

本期 MVP 允许简化为：

```text
Dify WebApp
    ↓
Dify Chatflow
    ↓
FastAPI
    ↓
MySQL
```

Vue 前端如果时间不足，可只做基础管理页面，核心演示优先用 Dify WebApp + Swagger。

## 5.3 模块职责

| 组件 | 职责 |
|---|---|
| Dify | 意图识别、对话生成、RAG 知识检索、工作流编排 |
| FastAPI | 业务接口、数据读写、业务规则校验、Dify 工具 API |
| MySQL | 保存用户、课程、活动、客户、学生服务、报告数据 |
| Vue 前端 | 后台管理页面、报告展示、业务表单（E 负责） |
| Redis | 本期可选，用于任务状态和缓存 |
| MinIO | 本期可选，用于客户资料文件存储 |
| Milvus | 本期可选，用于自建向量检索 |

## 5.4 主链路设计

必须优先跑通以下链路：

```text
用户在 Dify WebApp 输入：有什么语言培训课程？
        ↓
Dify 意图识别：课程查询
        ↓
Dify HTTP 节点调用：GET /api/v1/courses
        ↓
FastAPI 查询 MySQL 中的 course_project 表
        ↓
FastAPI 返回课程列表 JSON
        ↓
Dify 根据 JSON 组织自然语言回复
        ↓
用户看到课程推荐
```

这条链路是项目的"总装联调主线"。由成员 B 负责开发接口，成员 A 协助 Dify 联调环境。

## 5.5 用户登录认证设计（新增 V2.1）

### 5.5.1 设计思路

系统采用**统一登录入口 + 按身份分表存储扩展信息**的模式：

```text
┌─────────────────────────────────────────────────────────────┐
│                    用户登录流程                               │
│                                                             │
│  前端登录页                                                   │
│      ↓ POST /api/v1/auth/login {username, password}         │
│  FastAPI 验证账号密码                                         │
│      ↓ 查询 sys_user 表                                      │
│  根据 user_type 加载对应扩展信息                               │
│      ├── student  → 查 student_info 表                       │
│      ├── employee → 查 employee_info 表                      │
│      └── admin    → 仅 sys_user                             │
│      ↓ 生成 JWT Token（含 user_id, user_type, role_id）      │
│  返回 {token, user_info}                                     │
│      ↓                                                      │
│  前端存储 Token → Axios 拦截器自动附加 Header                 │
│      ↓                                                      │
│  后续请求携带 Authorization: Bearer <token>                  │
│      ↓                                                      │
│  FastAPI 依赖注入解析当前用户身份                              │
└─────────────────────────────────────────────────────────────┘
```

### 5.5.2 按身份分表存储扩展信息

| 身份 | user_type | 扩展信息表 | 存储字段 | 说明 |
|------|-----------|-----------|---------|------|
| 学生 | `student` | `student_info`（已有） | student_no, school, major, grade, abroad_country, class_teacher_id, enroll_date | 与 sys_user 通过 user_id 一对一关联 |
| 员工 | `employee` | `employee_info`（**新增**） | employee_no, position, department, hire_date, manager_id | 与 sys_user 通过 user_id 一对一关联 |
| 管理员 | `admin` | — | 直接使用 sys_user 的 department 等字段 | 不需要额外扩展表 |

### 5.5.3 新增数据库表 DDL

```sql
-- 员工扩展信息表（新增 V2.1）
DROP TABLE IF EXISTS `employee_info`;
CREATE TABLE `employee_info` (
    `id`                BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键',
    `user_id`           BIGINT UNSIGNED NOT NULL                COMMENT '关联 sys_user.id',
    `employee_no`       VARCHAR(32)     DEFAULT NULL            COMMENT '工号',
    `position`          VARCHAR(64)     DEFAULT NULL            COMMENT '职位（如：留学顾问、班主任、部门经理）',
    `department`        VARCHAR(128)    DEFAULT NULL            COMMENT '所属部门',
    `hire_date`         DATE            DEFAULT NULL            COMMENT '入职日期',
    `manager_id`        BIGINT UNSIGNED DEFAULT NULL            COMMENT '直属上级ID（关联 sys_user.id）',
    `status`            ENUM('active','resigned','suspended') NOT NULL DEFAULT 'active' COMMENT '在职状态',
    `create_time`       DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `update_time`       DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_user_id` (`user_id`),
    UNIQUE KEY `uk_employee_no` (`employee_no`),
    KEY `idx_manager_id` (`manager_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='员工扩展信息表';

-- 登录日志表（新增 V2.1，可选 P1）
DROP TABLE IF EXISTS `login_log`;
CREATE TABLE `login_log` (
    `id`                BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键',
    `user_id`           BIGINT UNSIGNED NOT NULL                COMMENT '用户ID',
    `login_ip`          VARCHAR(64)     DEFAULT NULL            COMMENT '登录IP',
    `login_time`        DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '登录时间',
    `user_agent`        VARCHAR(512)    DEFAULT NULL            COMMENT '浏览器UA',
    `status`            ENUM('success','failed') NOT NULL       COMMENT '登录状态',
    `fail_reason`       VARCHAR(255)    DEFAULT NULL            COMMENT '失败原因',
    PRIMARY KEY (`id`),
    KEY `idx_user_id` (`user_id`),
    KEY `idx_login_time` (`login_time`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='登录日志表';
```

### 5.5.4 JWT Token 设计

| 配置项 | 值 | 说明 |
|-------|-----|------|
| 算法 | HS256 | 对称加密，适合 MVP |
| 过期时间 | 24 小时 | `ACCESS_TOKEN_EXPIRE_HOURS=24` |
| Secret Key | 环境变量 `JWT_SECRET_KEY` | 至少 32 位随机字符串 |
| Token Payload | `{sub: user_id, user_type: "student/employee/admin", role_id: N, exp: timestamp}` | 最小必要信息 |

# 六、数据库设计流程

## 6.1 数据库设计目标

数据库设计要先于后端接口开发完成。表结构不稳定，会导致 ORM、API、前端字段和 Dify 节点全部反复修改。

**分工**：成员 A 在 Day1 上午主导数据库初始化（db_init.sql 执行），各模块负责人在各自模块中创建对应的 ORM Model 和种子数据。

## 6.3 表设计规范

每张表建议统一包含：

```text
id              INT PRIMARY KEY AUTO_INCREMENT
create_time     DATETIME DEFAULT CURRENT_TIMESTAMP
update_time     DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
status          VARCHAR(32)  -- 状态字段
```

命名建议：

1. 表名使用小写加下划线。
2. 字段名使用小写加下划线。
3. 状态字段使用英文枚举值，避免随意使用中文状态。
4. 外键字段统一以 `_id` 结尾。
5. 时间字段统一使用 `DATETIME`。
6. 大文本字段使用 `TEXT`。
7. 结构化 AI 结果可以使用 `JSON` 或 `TEXT` 存储。

# 九、Dify 工作流开发流程

## 9.1 Dify 开发目标

Dify 在本项目中不是孤立聊天机器人，而是业务流程编排中心。它需要：

1. 识别用户意图。
2. 判断是否需要调用 FastAPI。
3. 调用接口获取业务数据。
4. 将业务数据组织成自然语言。
5. 在必要时收集用户信息并提交表单类接口。

## 9.2 必做工作流（按模块归属）

### 10.2.1 前端登录流程

```text
用户访问任意页面
      ↓
router.beforeEach 检查是否有 Token
      ├── 无 Token → 跳转 /login
      └── 有 Token → 检查 Pinia auth store 是否有用户信息
            ├── 无用户信息 → 调用 GET /api/v1/auth/me 获取
            │     ├── Token 有效 → 存入 store → 放行
            │     └── Token 过期(401) → 清除 Token → 跳转 /login
            └── 有用户信息 → 放行
```

### 10.2.2 Token 管理要点

| 操作 | 说明 |
|------|------|
| 登录成功 | 将 `access_token` 存入 `localStorage`，用户信息存入 Pinia `authStore` |
| Axios 请求拦截器 | 自动从 `localStorage` 读取 Token，附加到 `Authorization: Bearer <token>` Header |
| Axios 响应拦截器 | 收到 401 → 清除 Token → 跳转登录页 |
| 退出登录 | 清除 `localStorage` Token + 清空 `authStore` → 跳转登录页 |
| 路由守卫 | `router.beforeEach` 检查登录状态，白名单路由（`/login`）放行 |

# 十五、常见问题与应对

## 15.1 环境问题

| 问题 | 症状 | 解决方案 |
|---|---|---|
| Dify HTTP 节点无法访问 FastAPI | 返回 Connection Refused | Dify 容器内使用 `host.docker.internal:8000` 代替 `localhost:8000` |
| CORS 跨域 | 前端调 FastAPI 被拦截 | A 在 main.py 添加 CORSMiddleware, allow_origins=["*"] |
| 数据库时区 | 存的时间和查询时间差 8 小时 | MySQL 设置 `time_zone='+08:00'`，使用 DATETIME |
| Dify API Key 错误 | 返回 401 | 去 Dify 后台重新生成 Key，确认 Bearer 格式 |
| Python 依赖冲突 | pip install 失败 | 使用虚拟环境 venv，严格按 requirements.txt |

## 15.2 开发问题

| 问题 | 解决方案 |
|---|---|
| 接口字段名不一致 | 严格按照 API 文档定义，Swagger 可自动校验 |
| 业务规则遗漏 | 参考开发引导.md 中的"注意事项"，每实现一个接口对照检查 |
| AI 编造数据 | Prompt 中强制约束"只依据接口返回数据回答" |
| 并发报名超额 | 使用数据库事务 + SELECT FOR UPDATE 或乐观锁 |
| 模块间依赖阻塞 | 优先完成自己模块的独立部分，联调时再对接 |
| JWT Token 过期 | 前端响应拦截器检测 401，自动清除 Token 跳转登录页 |
| 不同身份登录返回不同数据 | `/auth/me` 根据 `user_type` 查询对应扩展表（student_info / employee_info），admin 不返回 extended_info |
| 前端路由守卫未登录跳转 | `router.beforeEach` 检查 localStorage 中的 Token，白名单仅含 `/login` |

# 十六、Dify 与 FastAPI 边界划分

| 能力 | 应由谁负责 | 原因 |
|---|---|---|
| 意图识别 | Dify | LLM 擅长语义理解 |
| 自然语言回复 | Dify | LLM 核心能力 |
| 业务数据查询 | FastAPI | 数据库操作、SQL 优化 |
| 数据入库 | FastAPI | 事务、校验、业务规则 |
| 并发控制 | FastAPI | 数据库层面控制 |
| 用户登录认证 | FastAPI | JWT Token 签发/验证、密码哈希、身份分表查询 |
| 前端路由守卫 | Vue 前端 | 前端拦截未登录请求，保护页面 |
| 权限校验 | FastAPI | 后端安全边界（基于 JWT Payload 中的 user_type） |
| 报告文本生成 | Dify | LLM 擅长文本创作和分析 |
| 报告数据聚合 | FastAPI | 多表联查、数据统计 |

# 十七、结论

本项目 3 天内完成的关键不是功能数量，而是主链路质量。

核心原则：

1. **按模块分配**：每人负责一个完整模块，有独立可交付成果。
2. **先闭环，再完善**：每个模块先跑通最简链路，再加功能。
3. **并行推进**：B/C/D 三个模块完全并行，互不等待。
4. **数据驱动**：先用种子数据验证接口，再接入 AI。

最终必须能稳定演示：

```text
用户打开前端 → 跳转登录页 → 输入账号密码
→ POST /api/v1/auth/login → 返回 JWT Token + 身份信息
→ 前端存储 Token → 路由守卫放行 → 进入首页
→ 根据身份展示不同功能入口
→ 用户提问（Dify WebApp 或前端聊天窗口）
→ Dify 判断意图
→ 调用 FastAPI（携带 Token 的业务接口）
→ FastAPI 查询 MySQL
→ 返回业务数据
→ Dify 生成自然语言回复
→ 前端或 Dify WebApp 展示
```

只要这条链路稳定，答辩时就能清晰说明项目架构、技术选型、业务价值和后续扩展方向。每个成员都能展示自己负责的模块从接口到 Dify 联调到演示的完整闭环。
