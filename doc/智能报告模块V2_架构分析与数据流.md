### 各层核心职责一句话

| 层 | 核心职责 | 输入 | 输出 |
|----|---------|------|------|
| 路由层 | HTTP 请求准入、JWT 校验、参数验证 | HTTP Request | HTTP Response |
| 编排层 | 任务状态机、幂等控制、串起全流程 | report_type + 周期 + 过滤条件 | ReportGeneration 对象 |
| 注册表 | 报告类型元数据、聚合器映射 | report_type 字符串 | ReportDefinition |
| 聚合层 | SQL 取数 + 指标计算 | 数据库事实表 | AggregatedReport (content + snapshot + 质量) |
| 规则引擎 | 纯函数确定性计算 | 业务参数 | ApplicationRiskResult / ChannelROIMetrics / ... |
| AI解释层 | 自然语言解释（Dify/本地） | 聚合数据 + Schema 定义 | 补充了 summary/explanation 的 content |
| 渲染层 | HTML 模板渲染 | 结构化 JSON content | HTML 字符串 |
| 数据层 | ORM 映射、持久化 | Python 对象 | MySQL 行 |

## 3. 报告生成核心链路（完整 15 步数据流）

以 **POST /api/v1/reports/generate**（手动生成）为例，下面是完整的数据流：

```
┌─────────────────────────────────────────────────────────────────────────┐
│ Step 1 · 前端发起请求                                                    │
│                                                                         │
│  POST /api/v1/reports/generate                                          │
│  Header: Authorization: Bearer <JWT>                                    │
│  Header: Idempotency-Key: abc-123  (可选，防重复提交)                     │
│  Body: {                                                                │
│    "report_type": "application_risk",                                   │
│    "report_title": "2026年第28周申请风险报告",                            │
│    "period_start": "2026-07-06",                                        │
│    "period_end": "2026-07-12",                                          │
│    "filters": {"owner_id": 5}                                           │
│  }                                                                      │
└────────────────────────────┬────────────────────────────────────────────┘
                             │ HTTP POST
                             ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ Step 2 · JWT 认证中间件                                                  │
│                                                                         │
│  文件：utils/auth.py → get_current_user()                               │
│  解析 Authorization Header → 验证签名 → 查数据库 → CurrentUser 对象      │
│                                                                         │
│  输出：CurrentUser {                                                     │
│    id: 5,                                                               │
│    username: "zhangsan",                                                │
│    role_code: "manager",                                                │
│    user_type: "employee"                                                │
│  }                                                                      │
└────────────────────────────┬────────────────────────────────────────────┘
                             │ Depends(get_current_user)
                             ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ Step 3 · 请求体 Schema 校验                                              │
│                                                                         │
│  文件：routers/report.py → generate_report_endpoint()                   │
│        schemas/report.py → ReportGenerateRequest                        │
│                                                                         │
│  校验 report_type 非空、report_title 1-255字符、                          │
│  字段别名兼容：前端传 "title" 自动映射为 "report_title"                    │
│                                                                         │
│  → 输出：ReportGenerateRequest 对象                                      │
└────────────────────────────┬────────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ Step 4 · 查注册表获取报告定义                                            │
│                                                                         │
│  文件：services/reporting/registry.py → get_report_definition()          │
│                                                                         │
│  输入："application_risk"                                                │
│  输出：ReportDefinition {                                                │
│    report_type: "application_risk",                                     │
│    label: "申请风险报告",                                                 │
│    schema_version: 2,                                                   │
│    content_model: ApplicationRiskContent,                               │
│    aggregator_name: "aggregate_application_risk",                       │
│    template_name: "application_risk.html",                              │
│    allowed_roles: ("admin","manager","employee","team_leader"),          │
│    default_period_rule: "previous_week",                                │
│  }                                                                      │
└────────────────────────────┬────────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ Step 5 · 角色权限校验                                                    │
│                                                                         │
│  文件：utils/auth.py → ensure_report_permission()                       │
│  检查 current_user.role_code ∈ allowed_roles                             │
│  不通过 → 403 Forbidden                                                 │
│  学生角色 → 完全禁止访问管理报告                                          │
└────────────────────────────┬────────────────────────────────────────────┘
                             │ 权限通过
                             ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ Step 6 · 创建任务记录（写数据库）                                         │
│                                                                         │
│  文件：services/reporting/orchestrator.py → create_report_task()        │
│                                                                         │
│  操作：                                                                  │
│  1. resolve_period() → 如果前端没传日期，按 default_period_rule 推导     │
│  2. build_idempotency_key() → 有 Idempotency-Key 则生成                  │
│     "manual:abc-123"                                                    │
│  3. 用幂等键查重 → 已有记录则直接返回（防止重复提交）                     │
│  4. 写入 report_generation 表：                                          │
│     - report_type = "application_risk"                                  │
│     - status = "pending"                                                │
│     - schema_version = 2                                                │
│     - trigger_source = "manual"                                         │
│     - generated_by = 5                                                  │
│     - request_filters = {"owner_id": 5}                                 │
│     - data_quality = {"level":"ok", "warnings":[],                      │
│                       "data_source":"database"}                         │
│  5. db.commit() → 获得 report.id = 1                                    │
│                                                                         │
│  输出：ReportGeneration 对象 (id=1, status="pending")                    │
└────────────────────────────┬────────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ Step 7 · 注册后台任务                                                    │
│                                                                         │
│  文件：routers/report.py                                                │
│  background_tasks.add_task(generate_report_async, report.id)            │
│                                                                         │
│  注意：这里只注册不执行，后台任务在响应返回后启动                          │
└────────────────────────────┬────────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ Step 8 · 接口返回 202 Accepted                                          │
│                                                                         │
│  HTTP 202                                                               │
│  Body: {                                                                │
│    "id": 1,                                                             │
│    "report_type": "application_risk",                                   │
│    "report_title": "2026年第28周申请风险报告",                           │
│    "status": "pending",                                                 │
│    "schema_version": 2,                                                 │
│    "trigger_source": "manual",                                          │
│    "generated_by": 5,                                                   │
│    "create_time": "2026-07-10T10:00:00"                                 │
│  }                                                                      │
│                                                                         │
│  前端收到 202 后：显示"报告生成中"，用 report_id 轮询 GET /reports/{id}  │
└─────────────────────────────────────────────────────────────────────────┘
                             │
        ┌────────────────────┘
        │ (响应已返回给前端，以下在后台线程执行)
        ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ Step 9 · 后台任务启动                                                    │
│                                                                         │
│  文件：services/reporting/orchestrator.py → generate_report_async()     │
│                                                                         │
│  关键：BackgroundTasks 在独立线程运行，必须自己创建数据库 Session          │
│  db = SessionLocal()  ← 不复用请求线程的 Session                         │
└────────────────────────────┬────────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ Step 10 · 开始生成（状态：pending → generating）                            │
│                                                                             │
│  文件：services/reporting/orchestrator.py → generate_report()               │
│                                                                             │
│  操作：                                                                      │
│  1. db.query(ReportGeneration).filter_by(id=1) → report 对象                  │
│  2. report.status = "generating"                                             │
│  3. report.started_time = datetime.now()                                     │
│  4. db.commit()                                                              │
│                                                                              │
│  然后按顺序调用：                                                             │
│  ┌──────────────────────────────────────────────────────────────────────┐   │
│  │ aggregate_report()  →  enrich_content_with_ai()  →  render_report() │   │
│  └──────────────────────────────────────────────────────────────────────┘   │
└────────────────────────────┬────────────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ Step 11 · 数据聚合（SQL 查询 + 规则计算）                                     │
│                                                                             │
│  文件：services/reporting/aggregators.py → aggregate_report()               │
│                                                                             │
│  调用链：                                                                    │
│  aggregate_report(db, "application_risk", start, end, filters)               │
│    → globals()["aggregate_application_risk"]                                 │
```

### 4.2 聚合器之间的复用关系

```
aggregate_weekly_summary (综合周报)
  ├── 调用 aggregate_customer_ops()    → customer_ops 的指标
  ├── 调用 aggregate_daily_summary()   → 日报汇总指标
  ├── 调用 aggregate_psych_weekly()    → 心理预警指标
  │     └── 内部调用 aggregate_service_sla()
  └── 调用 aggregate_complaint_weekly()→ 投诉处理指标
        └── 内部调用 aggregate_service_sla()

aggregate_customer_ops (客户经营)
  └── 调用 aggregate_sales_funnel()    → 复用漏斗指标

aggregate_psych_weekly (心理周报)
  └── 调用 aggregate_service_sla()     → 提取心理预警 SLA 部分

aggregate_complaint_weekly (投诉周报)
  └── 调用 aggregate_service_sla()     → 提取投诉 SLA 部分
```

设计意图：`service_sla` 作为基础聚合器，同时覆盖投诉和心理预警，被 `complaint_weekly` 和 `psych_weekly` 复用。`weekly_summary` 作为最顶层聚合器，汇总所有子报告的指标形成跨模块管理视图。


## 7. 三条触发路径对比

智能报告模块有三种触发报告生成的方式，它们**共用同一套编排层**，保证行为一致：

```
╔═══════════════════════════════════════════════════════════════════════════╗
║                        路径一：手动触发                                    ║
║                                                                           ║
║  前端 POST /api/v1/reports/generate                                       ║
║    │                                                                      ║
║    ├─ routers/report.py → generate_report_endpoint()                      ║
║    │   ├─ get_report_definition() → 查注册表                              ║
║    │   ├─ ensure_report_permission() → 权限校验                            ║
║    │   └─ create_report_task()  → 写 DB (status=pending)                  ║
║    │   └─ background_tasks.add_task(generate_report_async, report_id)     ║
║    │                                                                      ║
║    ├─ 返回 202 (任务已创建)                                                ║
║    │                                                                      ║
║    └─ [后台线程] generate_report_async(report_id)                         ║
║        └─ SessionLocal() → generate_report() → 聚合→AI→渲染→保存         ║
║                                                                           ║
║  幂等键：manual:{Idempotency-Key 请求头}                                   ║
║  (如果同一 Key 已有记录，直接返回已有记录，不重复创建)                      ║
╚═══════════════════════════════════════════════════════════════════════════╝

╔═══════════════════════════════════════════════════════════════════════════╗
║                        路径二：定时触发                                    ║
║                                                                           ║
║  独立进程 python report_scheduler.py                                      ║
║    │                                                                      ║
║    ├─ APScheduler BlockingScheduler 启动                                  ║
║    │   ├─ 读取 report_schedule 表 (enabled=1)                              ║
║    │   ├─ 为每个计划注册 CronTrigger 任务                                  ║
║    │   └─ coalesce=True (错过多次合并为一次)                               ║
║    │                                                                      ║
║    ├─ [到点触发] run_schedule_once(schedule_id)                           ║
║    │   ├─ SessionLocal()                                                  ║
║    │   ├─ 查 report_schedule 确认仍启用                                   ║
║    │   ├─ resolve_period() → 按 period_rule 计算周期                       ║
║    │   ├─ create_report_task(trigger_source="schedule")                   ║
║    │   │   └─ 幂等键 = schedule:{id}:{start}:{end}                        ║
║    │   ├─ generate_report(report.id, db)  ← 同步执行，不依赖FastAPI       ║
║    │   └─ 更新 schedule.last_status/last_error/last_run_time              ║
║    │                                                                      ║
║  幂等键：schedule:{schedule_id}:{period_start}:{period_end}               ║
║  (同一计划+同一周期不会重复生成)                                            ║
║                                                                           ║
║  调度器为什么独立进程？                                                    ║
║  - FastAPI 多 worker 不会各自启动调度器导致重复                            ║
║  - 调度器崩溃不影响 API 服务                                               ║
║  - 可以独立部署、监控、重启                                                ║
╚═══════════════════════════════════════════════════════════════════════════╝

╔═══════════════════════════════════════════════════════════════════════════╗
║                        路径三：重试触发                                    ║
║                                                                           ║
║  前端 POST /api/v1/reports/{report_id}/retry                              ║
║    │                                                                      ║
║    ├─ routers/report.py → retry_report_endpoint()                         ║
║    │   ├─ 查原始报告 (必须存在)                                            ║
║    │   ├─ 权限校验 (能访问原报告 + 有该类型权限)                            ║
║    │   └─ retry_report() → create_report_task(trigger_source="retry")     ║
║    │       ├─ 继承原报告的 type/周期/过滤条件                              ║
║    │       ├─ title 加 "（重试）" 后缀                                    ║
║    │       ├─ retry_of_report_id = 原报告ID                               ║
║    │       ├─ retry_count = 原报告.retry_count + 1                        ║
║    │       └─ 不生成幂等键 (每次重试都是独立新记录)                        ║
║    │                                                                      ║
║    └─ background_tasks.add_task(generate_report_async, new_report.id)     ║
║                                                                           ║
║  ★ 关键设计：重试创建新记录，不覆盖原失败记录                              ║
║    这样原失败原因完整保留，便于排查 "为什么第一次失败了"                    ║
╚═══════════════════════════════════════════════════════════════════════════╝
```

### 三条路径汇总对比

| 维度 | 手动触发 | 定时触发 | 重试触发 |
|------|---------|---------|---------|
| 入口 | `POST /generate` | APScheduler cron | `POST /{id}/retry` |
| trigger_source | `manual` | `schedule` | `retry` |
| 幂等键 | manual:{uuid} | schedule:{id}:{start}:{end} | 无（每次独立） |
| 执行方式 | BackgroundTasks 异步 | 同步（独立进程） | BackgroundTasks 异步 |
| DB Session | 两步：请求Session + 独立Session | 独立 SessionLocal | 两步：请求Session + 独立Session |
| 周期来源 | 请求参数 or 默认规则 | period_rule 推导 | 继承原报告 |
| 失败处理 | 记录 error_message | 记录 error_message + schedule.last_error | 创建新记录，通过 retry_of_report_id 关联 |
| extends 原报告 | — | — | 是，retry_of_report_id + retry_count |

```
  └── services/reporting/orchestrator.py (re-export)
  └── services/reporting/aggregators.py  (re-export)
```

**依赖原则**：
- `registry.py` ← `schemas.py`（注册表依赖内容模型定义）
- `aggregators.py` ← `registry.py` + `rules.py`（聚合器查定义 + 调规则）
- `ai_generator.py` ← `registry.py`（AI 层需要知道 Schema 定义来校验）
- `orchestrator.py` ← 所有子模块（编排层是唯一串联所有模块的地方）
- `routers/*.py` ← `orchestrator.py` + `registry.py`（路由层只接触编排和注册表）
- 最深依赖链：`router → orchestrator → aggregator → rules`（4 层）

### 8.2 V2 改造的核心数据流原则总结

```
┌─────────────────────────────────────────────────────────────────────────┐
│  原则 1：数字由 SQL 和规则引擎计算，AI 只做解释                           │
│                                                                         │
│  aggregators.py ──出数字──→ rules.py ──出结构化结果──→ ai_generator.py  │
│  (SQL 查询事实表)          (纯函数确定性计算)        (只改 summary 和     │
│                                                      explanation)       │
│  验证方式：aggregated_data_snapshot 保存聚合快照，可随时回溯             │
├─────────────────────────────────────────────────────────────────────────┤
│  原则 2：每类报告独立 Schema，注册表驱动扩展                              │
│                                                                         │
│  新增报告只需三步：                                                      │
│  1. schemas.py 中新增 Content 模型 (如 ApplicationRiskContent)          │
│  2. aggregators.py 中新增聚合器函数 (如 aggregate_application_risk)     │
│  3. registry.py 中注册一行 (report_type → 聚合器/模板/权限/周期)        │
│                                                                         │
│  不需要改 orchestrator、router、renderer 任何代码                        │
├─────────────────────────────────────────────────────────────────────────┤
│  原则 3：任务生命周期完整记录，失败可追溯可重试                            │
│                                                                         │
│  status 流转：pending → generating → completed                          │
│                                  ↘ failed → (retry) → 新 pending        │
│                                                                         │
│  每次尝试一条独立记录：                                                   │
│  - 原始任务：retry_count=0, retry_of_report_id=NULL                     │
│  - 第一次重试：retry_count=1, retry_of_report_id=原始ID                  │
│  - 第二次重试：retry_count=2, retry_of_report_id=第一次重试ID            │
│                                                                         │
│  这样形成完整重试链路，可以追溯"哪次尝试因为什么原因失败"                  │
├─────────────────────────────────────────────────────────────────────────┤
│  原则 4：AI 输出必须通过 Pydantic Schema 校验，两次机会                   │
│                                                                         │
│  enrich_content_with_ai() 返回后立即执行：                               │
│  definition.content_model.model_validate(merged)                         │
│                                                                         │
│  第一次失败 → 带 validation_error 信息重新调 Dify                       │
│  第二次仍失败 → 任务进入 failed，error_code = "REPORT_GENERATION_FAILED" │
│                                                                         │
│  这保证了前端拿到的 report_content 一定是结构合法的                       │
├─────────────────────────────────────────────────────────────────────────┤
│  原则 5：事实表与报告一一对应，数据来源可追溯                             │
│                                                                         │
│  每类新增报告都有对应的数据库事实表：                                     │
│  application_risk  → application_material_item                          │
│  sales_funnel      → crm_lead + crm_lead_status_history                 │
│  channel_roi       → marketing_channel_cost + contract + payment         │
│  service_sla       → student_feedback_ticket + student_psych_alert      │
│  action_closure    → report_action                                      │
│                                                                         │
│  MetricTrace 记录每个指标的来源表、公式和过滤条件                         │
│  DataQuality 标记数据完整性和来源（database / local / mock）              │
│                                                                         │
│  "这个数字从哪来的？" → 看 metric_traces                                │
│  "这个数据可靠吗？" → 看 data_quality.level                              │
└─────────────────────────────────────────────────────────────────────────┘
```
