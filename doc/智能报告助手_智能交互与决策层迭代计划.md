# 智能报告助手：智能交互与决策层迭代实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development`（推荐）或 `superpowers:executing-plans`，按任务逐项实施。所有步骤使用复选框跟踪。
>
> **执行前置规则：** 先阅读项目根目录 `CLAUDE.md`、`智能报告模块V2_十份报告深度解析.md`、智能报告相关代码和测试。若本计划与 `CLAUDE.md` 冲突，以 `CLAUDE.md` 为准，并在编码前列出冲突点和调整方案。

**目标：** 在不推翻现有十类报告、聚合器、规则引擎、编排器、渲染器、权限和前端契约的前提下，新增一个“智能交互与决策层”，让管理者能够用自然语言生成、查询、追问、比较报告，并在人工确认后形成行动闭环。

**架构：** 现有报告编排链路继续作为唯一事实生产链路。新增智能层只负责自然语言意图识别、参数规范化、报告工具选择、多轮上下文、证据化回答和候选行动建议；所有业务数字仍由 SQL、Aggregator 和 Rules 计算，所有数据库写操作仍由受控 Python 服务执行。

**技术栈：** Python 3.12、FastAPI、Pydantic v2、SQLAlchemy、现有报告 Registry/Aggregator/Rules/Orchestrator、兼容 OpenAI 协议的大模型 SDK、pytest。第一阶段不引入 LangGraph、向量数据库或多 Agent。

---

## 0. 全局约束

1. 不修改现有十类报告的业务定义、核心指标、风险评分、ROI/CPL/CAC/SLA 计算口径。
2. 不让 LLM 直接生成、修改或覆盖任何业务数字。
3. 不让 LLM 拼接 SQL，不向 LLM 暴露 SQLAlchemy Session。
4. 不让 LLM 输出未经校验的数据直接写入数据库。
5. 原 `POST /api/v1/reports/generate`、报告列表、详情、重试、定时任务接口继续工作。
6. 新智能入口必须复用现有 `registry → aggregator → rules → orchestrator → renderer` 链路。
7. 心理预警场景只使用聚合后的匿名数据，不传输学生原文、诊断文本或不必要的个人信息。
8. 所有写操作必须满足：权限校验 → Pydantic 校验 → 用户明确确认 → 幂等控制 → 服务层写入。
9. 第一阶段优先采用 Structured Output + 受控 Tool Calling；不为展示技术而引入复杂 Agent。
10. 公共文件默认不修改。新增路由优先通过现有 `routers/report.py` 内部挂载，避免修改 `main.py` 或全局路由注册。
11. 未确认实际代码签名之前，不得假设函数名称、返回结构或字段存在。
12. 每个迭代必须可独立发布、独立回滚，不得一次性大范围重构。

# 一、现状基线与目标边界

## 1.1 保留的现有能力

以下能力继续作为系统事实基础，不重写：

- `REPORT_REGISTRY`：报告类型、内容 Schema、聚合器、默认周期、权限。
- `aggregators.py`：数据库取数和结构化聚合。
- `rules.py`：风险分、ROI、转化率、SLA 等确定性规则。
- `orchestrator.py`：任务创建、周期解析、幂等、生成、重试、持久化。
- `schemas.py`：十类报告内容模型、`ReportEnvelope`、`DataQuality`、`MetricTrace`。
- `renderer.py`：HTML 渲染。
- `routers/report.py`：现有报告接口。
- 前端报告列表、生成、详情、轮询、重试、调度、行动项页面。
- `report_action` 的正式行动闭环。
- 数据质量、指标追溯、权限、Schema 版本兼容。

## 1.2 新增的智能层职责

```text
自然语言输入
→ 意图识别
→ 报告类型映射
→ 时间和筛选参数规范化
→ 置信度与澄清判断
→ 调用受控报告工具
→ 获取确定性数据
→ 基于证据生成回答
→ 返回引用、假设和建议追问
→ 用户确认后调用行动写入工具
```

## 1.3 明确不做

- 通用 NL2SQL。
- 让模型自行分析数据库 Schema。
- 多 Agent 协作。
- 知识图谱。
- 向量数据库/RAG。
- 机器学习风险预测。
- 自动修改风险规则。
- 自动创建正式行动项。
- 自动向外部渠道发送通知。
- 替换现有前端全部报告页面。
- 删除现有 Dify 代码；Dify 到纯 Python 的迁移应作为独立重构任务处理。

# 四、模糊需求与安全规则

## 4.1 默认执行并明示假设

- “最近申请风险”：按注册表默认周期或当前周期截至今天。
- “老板能看懂的周报”：`weekly_summary + management_summary`。
- “哪个渠道最不划算”：`channel_roi`，按确定性 ROI 升序，并显示数据质量。

## 4.2 必须追问

- 实体不唯一。
- “第一个”无法映射到上下文实体。
- 时间歧义会显著影响口径且无安全默认值。
- 多个互相冲突的意图。
- 写操作。
- 意图置信度过低。
- 不存在的报告或字段。
- 用户要求当前数据无法证明的因果结论。

## 4.3 建议置信度

```text
confidence >= 0.80：直接执行，明示假设
0.55 <= confidence < 0.80：只读执行或在正式生成前追问
confidence < 0.55：不调用业务工具，返回具体澄清问题
```

阈值配置化，并由评估集校准。

## 4.4 安全底线

- 用户输入不能直接进入 SQL。
- LLM 不能访问 DB Session。
- LLM 只能调用预注册工具。
- 每个工具独立鉴权。
- 工具参数和结果经过 Pydantic。
- 回答中的数字必须在工具结果中找到。
- 数据质量为 `warning/degraded/failed` 时必须表达限制。
- 心理数据只允许聚合、匿名结果。
- 写操作需要用户确认、确认令牌和幂等键。
- 未经证据不得使用“导致、证明、必然因为”。

## Iteration 3：周期对比和受控跨报告分析

**目标：** 支持周期变化、多个报告工具和相关性分析。

### Task 3.1：指标白名单

**创建：**

- `services/reporting/assistant/metric_catalog.py`
- `tests/test_report_assistant_metric_catalog.py`

每个报告定义：

- 可比较指标。
- JSON 路径。
- 数值类型。
- 是否允许变化率。
- 缺失值语义。
- 敏感等级。

禁止模型自由指定 JSONPath。

### Task 3.2：周期对比工具

**新增接口：**

```python
def compare_report_metrics(
    *,
    report_type: str,
    current_period: ResolvedPeriod,
    comparison_period: ResolvedPeriod,
    metric_names: list[str],
    current_user: CurrentUser,
    db: Session,
) -> AssistantToolResult:
    ...
```

规则：

- 差值与变化率由 Python 计算。
- 分母为 0 时变化率为 `None`。
- 两周期口径必须一致。
- `failed` 不下结论。
- `warning/degraded` 附限制。

### Task 3.3：跨报告计划

一轮工具调用建议上限 3 个；仅允许预注册组合；每个子报告独立鉴权；敏感报告不得间接泄露。

### Task 3.4：因果保护

回答固定分区：

- 已确认事实。
- 相关信号。
- 可能解释。
- 无法确认。
- 建议验证。

**Iteration 3 验收：**

- 本周与上周申请风险可比较。
- 渠道 ROI 相邻周期可比较。
- 投诉与 SLA 问题不产生伪因果。
- 多报告逐个鉴权。
- 0、None、warning、degraded 边界测试通过。

## Iteration 4：候选行动、人工确认和主动提醒

**目标：** 洞察转为候选行动，人工确认后进入现有 `report_action`；增加确定性主动提醒。

### Task 4.1：候选行动

```python
class ReportActionCandidate(BaseModel):
    candidate_id: UUID
    source_report_id: int
    source_report_type: str
    source_entity_type: str
    source_entity_id: str
    risk_code: str
    title: str
    description: str
    suggested_owner_role: str | None
    suggested_due_time: datetime | None
    priority: Literal["urgent", "high", "medium", "low"]
    evidence: list[EvidenceItem]
    requires_confirmation: Literal[True] = True
```

约束：

- `risk_code` 和证据由 Python 填充。
- AI 不指定真实员工 id。
- 候选项不写数据库。
- 时间和优先级由 Python 校验。

### Task 4.2：确认写入

建议接口：

```http
POST /api/v1/reports/assistant/actions/confirm
```

包含：

- conversation id。
- candidate ids。
- confirmation token。
- idempotency key。

要求：

- 令牌绑定候选内容摘要。
- 二次鉴权。
- 调用现有行动服务。
- 重复确认不重复创建。
- 部分失败逐条返回。

### Task 4.3：主动事件

第一版只做固定阈值和状态变化，不宣称机器学习异常检测：

- 申请风险升级。
- 临近截止且材料未齐。
- 投诉即将超 SLA。
- 心理高风险即将超跟进时限。
- 渠道 ROI 连续为负。
- 停滞线索增加。
- 行动项逾期。
- 相同 risk_code 重复。
- 数据质量恶化。
- 报告生成失败。

要求：

- 复用独立 scheduler，不在 API worker 启动新调度器。
- 使用稳定事件键防重复。
- 无现成通知体系时先提交设计，不擅自建设公共通知模块。
- AI 只归纳已触发事件，不创造事件。

**Iteration 4 验收：**

- 未确认时无 DB 写入。
- 确认后进入现有行动闭环。
- 重复确认幂等。
- 主动提醒有触发条件、当前值、阈值和证据。
- 心理提醒无敏感原文。
- 不新增第二套 scheduler。

# 九、最终验收

## 产品

1. 管理者无需知道报告编码即可自然语言发起核心报告。
2. 模糊时间有安全默认值和明确假设。
3. 申请风险支持至少四轮追问。
4. 指标和风险可以解释来源、公式和周期。
5. 支持周期对比和受控跨报告分析。
6. 数据不足时拒绝错误结论。
7. AI 建议确认后才写入 `report_action`。
8. 原报告页面和接口不受影响。

## 技术

1. LLM 不直接访问 DB、不写 SQL、不计算业务指标。
2. 工具参数和结果全部 Pydantic 校验。
3. 回答数字全部可定位到工具证据。
4. 权限、数据质量、幂等测试通过。
5. LLM 不可用时有降级。
6. 原 V2 测试 100% 通过。
7. 开关关闭后恢复原行为。

## 灰度建议门槛

- 报告类型识别准确率 ≥ 95%。
- 时间解析准确率 ≥ 95%。
- 工具选择准确率 ≥ 95%。
- 权限拦截准确率 = 100%。
- 数字引用准确率 = 100%。
- 非法数字生成率 = 0。
- 多轮实体解析准确率 ≥ 95%。
- Structured Output 成功率 ≥ 98%。
