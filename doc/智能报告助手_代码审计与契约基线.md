## 一、CLAUDE.md 相关规则摘要

项目根目录 `CLAUDE.md` 与本轮审计直接相关的规则：

| 规则编号 | 内容 | 对本轮影响 |
|----------|------|-----------|
| §1.1 项目 AI 架构总览 | 客服 Agent 仍用 Dify；其余模块纯 Python LLM 直连 | 智能报告模块已纯 Python 化，智能层不新增 Dify 调用 |
| §1.1 关键边界 | 不允许新模块调用 `utils/dify_client.py` | 新增 assistant 模块不能引入 Dify 依赖 |
| §6.1 Dify 使用边界 | Dify 只能用于 chat 模块 | assistant 不属于 chat 模块，确认禁止 |
| §6.2 纯 Python LLM 调用规范 | 7 个步骤：Prompt 构建 → Client 封装 → JSON 解析 → Pydantic 校验 → 修复重试 → 数据脱敏 → 日志记录 | 智能层的 LLM 调用也须遵循此模式 |
| §6.3 LLM 配置命名空间 | `REPORT_LLM_*` vs `LLM_*` vs `DIFY_*` 三者隔离 | 智能层 LLM 调用应复用 `REPORT_LLM_*` 环境变量（或新增独立前缀 `REPORT_ASSISTANT_LLM_*`） |
| §6.4 AI/RAG 链路注释重点 | 每个 AI 功能标注链路位置 | 新增 assistant 代码必须标注链路位置 |
| §3 注释生成四段法 | 文件/接口/类/函数/逻辑块/返回值/测试注释要求 | 新增代码必须符合中文注释规范 |
| §5 FastAPI 分层注释重点 | routers/services/schemas/models/utils/tests 各层注释重点 | 新增文件须按层写注释 |

**关键结论**：智能层不引入 Dify，LLM 调用复用现有 `ReportLLMClient` 或基于同名 `openai` SDK 模式新增独立 Client；注释必须符合四段法。

## 二、当前智能报告完整调用链

### 2.1 报告生成主链路（同步 + 后台任务）

```text
┌─────────────────────────────────────────────────────────────────────┐
│ 触发入口（三选一）                                                      │
│ ├─ POST /api/v1/reports/generate  →  manual（幂等头 Idempotency-Key）    │
│ ├─ APScheduler cron 唤醒          →  schedule（power key: schedule:{id}:{start}:{end}）│
│ └─ POST /api/v1/reports/{id}/retry →  retry（新记录，retry_of_report_id 指向前身）│
└──────────────────────────┬──────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────────────┐
│ 编排层（orchestrator.py）                                              │
│                                                                     │
│ create_report_task()                                                 │
│   ├─ get_report_definition(report_type) → ReportDefinition             │
│   ├─ resolve_period(start, end, period_rule) → (date, date)            │
│   ├─ build_idempotency_key(...) → idempotency_key                      │
│   ├─ 幂等检查：db.query(idempotency_key=key).first()                     │
│   ├─ 创建 ReportGeneration(status="pending")                           │
│   └─ 返回 record → 接口返回 202                                      │
│                                                                     │
│ generate_report_async(report_id)  ← BackgroundTasks                   │
│   └─ generate_report(report_id, db)                                   │
│       ├─ status → "generating"                                        │
│       ├─ aggregate_report(db, report_type, start, end, filters)        │
│       ├─ enrich_content_with_ai(definition, title, period, content, dq)│
│       ├─ content_model.model_validate(content)  ←  再校验              │
│       ├─ render_report_html(definition, title, period, content, dq)     │
│       └─ status → "completed"  /  _mark_failed()                       │
└──────────────────────────┬──────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────────────┐
│ 聚合层（aggregators.py）                                              │
│                                                                     │
│ aggregate_report(db, report_type, start, end, filters)                │
│   ├─ get_report_definition → aggregator_name                           │
│   ├─ globals()[aggregator_name](db, start, end, filters)               │
│   └─ 返回 AggregatedReport(content, snapshot, data_quality)            │
│                                                                     │
│ 每个聚合器内部：                                                       │
│   ├─ _rows() / _scalar() → 安全 SQL 执行（可选表缺失→warning 而非崩溃）│
│   ├─ rules.py 纯函数 → 风险分/ROI/SLA/转化率                           │
│   └─ 组装 content + MetricTrace + 数据质量                             │
└──────────────────────────┬──────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────────────┐
│ AI 解释层（ai_generator.py）                                          │
│                                                                     │
│ enrich_content_with_ai(definition, title, period, content, dq)        │
│   ├─ REPORT_AI_MODE=local → _local_explanation()                       │
│   ├─ REPORT_AI_MODE=llm:                                               │
│   │   ├─ sanitize_report_data() → Python 层面字段白名单脱敏              │
│   │   ├─ build_chat_messages() → System + User Prompt                  │
│   │   ├─ ReportLLMClient.chat_completion() → 同步 + 指数退避重试         │
│   │   ├─ _parse_llm_json() → 裸JSON/代码块/内嵌JSON 三种格式            │
│   │   ├─ _merge_content() → 只允许 summary/explanation 覆盖             │
│   │   ├─ content_model.model_validate() → Pydantic 校验                │
│   │   └─ 失败 → _repair()（1次修复机会）                               │
│   └─ 返回完整 content dict                                            │
└──────────────────────────┬──────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────────────┐
│ 渲染层（renderer.py）                                                 │
│                                                                     │
│ render_report_html(definition, title, period, content, dq)            │
│   ├─ 有模板文件 → 轻量字符串替换                                       │
│   └─ 无模板文件 → 内置 HTML 骨架 fallback                                │
└─────────────────────────────────────────────────────────────────────┘
```

```

GET  /api/v1/reports/{id}          → 详情（行级权限检查）
POST /api/v1/reports/{id}/actions  → 创建行动项
GET  /api/v1/reports/{id}/actions  → 查询行动项列表
PATCH /api/v1/reports/actions/{id} → 更新行动项
```

### 2.3 路由注册机制

```text
routers/__init__.py → register_routers(app)
  └─ report_router → prefix="/api/v1/reports", tags=["智能报告"]

所有路由在 routers/__init__.py 集中注册。main.py 只调用 register_routers(app)。
新增智能路由时，应在 routers/__init__.py 的 register_routers() 中添加一行，
而非修改 main.py。
```

## 五、权限、幂等、数据质量和追溯机制

### 5.1 权限控制

```text
层级 1：GET /types  →  学生角色直接 403
层级 2：POST /generate →  ensure_report_permission(user, definition.allowed_roles)
         ├─ ensure_management_user(user)   → 学生 403
         ├─ admin 直接通过
         └─ user.role_code not in allowed_roles → 403

层级 3：GET /{id}    →  _check_report_row_access(report, user)
         ├─ admin/manager/team_leader 可查看管理范围
         └─ 普通员工只能看 generated_by == user.id 的报告

层级 4：POST /{id}/actions → 复用层级3 + definition.allowed_roles
```

### 5.2 幂等机制

```python
# create_report_task() 中实现：
# 1. 手动接口：Idempotency-Key 请求头 → "manual:{key}"
# 2. 定时任务：schedule_id + period → "schedule:{id}:{start}:{end}"
# 3. 重试接口：不生成幂等键（每次新记录）

# 数据库层：uk_report_generation_idempotency UNIQUE INDEX
# 重复请求直接返回已有记录，不创建新任务
```

### 5.3 数据质量

```python
class DataQuality:
    level: Literal["ok", "warning", "degraded", "empty", "failed"]
    # 可选数据源缺失 → warning（不阻塞）
    # 周期内无数据 → empty（不阻塞，但内容为空）
    # 必需数据源缺失 → RuntimeError（阻断）
```

### 5.4 指标追溯

```python
class MetricTrace:
    metric_name: str
    source_tables: list[str]    # 事实表
    formula: Optional[str]      # 计算公式
    filters: dict[str, Any]     # 当时过滤条件

# 每个聚合器都在 content["metric_traces"] 中附带追溯信息
```

### 5.5 LLM 安全边界（已有）

- **数字保护**：`_merge_content()` 只允许 LLM 覆盖 `summary` 和 `explanation`
- **脱敏**：`sanitize_report_data()` + System Prompt 约束
- **修复不编造**：修复 Prompt 显式传入 `invalid_output` + `validation_error`
- **日志脱敏**：不记录 API Key 和完整 Prompt内容

## 十二、风险与待确认问题

### 12.1 高风险

| 风险 | 严重程度 | 说明 | 缓解措施 |
|------|---------|------|---------|
| LLM 编造数字 | 🔴 高 | 智能层回答若让 LLM 自由发挥可能编造业务数据 | 回答只能引用工具返回的数字；Python 后校验 |
| 权限绕过 | 🔴 高 | 智能接口若不独立鉴权，学生可能通过自然语言获取管理报告 | 每个工具调用前 `ensure_report_permission()` |
| 心理数据泄露 | 🔴 高 | LLM 可能从 `alert_status` 中推断并输出学生信息 | `sanitize_report_data()` + Prompt 双重约束 + 回答后校验 |
| 报告类型映射错误 | 🟡 中 | 自然语言可能误识别报告类型（如把"投诉"映射成 service_sla 而非 complaint_weekly） | 置信度 + 澄清策略 + 白名单约束 |

### 12.2 待确认

| # | 问题 | 建议 |
|---|------|------|
| 1 | 智能层是否需要独立的 LLM 配置（`REPORT_ASSISTANT_LLM_*`）还是复用 `REPORT_LLM_*`？ | 建议 Iteration 1 先复用，后续按需拆分 |
| 2 | /assistant/messages 接口是同步返回还是异步+轮询？（LLM 调用 + 报告生成可能 5-15s） | 建议 Iteration 1 同步，设合理超时；若体验差再改异步 |
| 3 | intent_parser 是否复用 `ReportLLMClient` 还是使用更小的专用模型？ | 建议先复用，通过 temperature 等参数调优 |
| 4 | 会话上下文跨请求回传，多大算大？（10 个 entity 引用？100 个？） | 建议上限 20 个 entity，超出裁剪最早 |
| 5 | 前端 Iteration 2 的对话面板放在哪个路由？（新建页面 vs 报告中心嵌入？） | 待前端审计后确认 |

| 7 | function calling vs Structured Output 选哪个？ | 建议先用 Structured Output 做意图识别（轻量、可控），工具调用先用 Python 确定性映射 |

### 12.3 低风险但需注意

| 项目 | 说明 |
|------|------|
| 报告生成是同步 + 后台任务混合 | 智能接口选同步会阻塞请求线程（LLM ~3-10s）；选后台+轮询则前端需适配 |
