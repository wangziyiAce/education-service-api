# 一、项目概述

## 1.1 项目背景

留学教育服务机构通常同时面对客户咨询、客户转化、员工业务管理、学生服务、投诉处理、心理关怀、经营分析等多类业务。传统人工处理方式存在以下问题：

1. 客户咨询重复度高，人工客服压力大。
2. 客户资料来源复杂，画像研判依赖人工经验。
3. 员工客户跟进、日报、业务查询分散，缺乏统一入口。
4. 学生请假、投诉、申请进度、心理关怀等服务缺少闭环管理。
5. 管理层需要周期性经营报告，但人工整理耗时。

因此，本项目建设一个 AI 驱动的教育服务系统，通过 Dify 提供对话理解、RAG 检索和工作流编排能力，通过 FastAPI 提供稳定的业务接口和数据处理能力，通过 MySQL 等存储系统保存核心业务数据。

项目整体分工采用“Dify 做大脑，FastAPI 做手脚”的模式：Dify 负责意图识别、对话生成、RAG 检索和工作流编排，FastAPI 负责业务数据 CRUD、Dify 工具 API、回调通知和状态管理。

## 1.2 项目名称

**教育服务系统（留学服务智能平台）**

## 1.3 项目定位

本项目是面向留学教育服务机构的 AI 驱动业务服务平台，基于：

```text
FastAPI + Dify + MySQL + Redis + MinIO + Milvus
```

实现客户咨询、客户研判、企业员工助手、学生智能服务和智能报告生成等核心能力。

## 1.4 建设目标

系统建设目标如下：

1. 建立统一的教育服务业务后台。
2. 支持客户资料上传、AI 画像研判和产品匹配推荐。
3. 支持客服 Agent 完成政策问答、课程推荐、活动报名等服务。
4. 支持企业员工通过智能助手管理客户、提交日报、查询业务数据。
5. 支持学生通过智能助手完成请假、投诉、申请进度查询、心理关怀等操作。
6. 支持管理层自动生成客户经营分析、日报汇总、心理周报、投诉处理周报。
7. 建立 Dify 与 FastAPI 的双向调用机制。

## 1.5 项目边界

### 本期实现范围

本期重点实现：

1. 客户研判基础流程。
2. 客服 Agent 核心场景。
3. CRM 意向客户与跟进管理。
4. 学生请假、投诉、心理预警基础闭环。
5. 智能报告生成基础能力。
6. Dify 调 FastAPI、FastAPI 调 Dify 双模式联调。

### 本期不重点实现

以下能力可作为后续扩展：

1. 完整 SaaS 多租户系统。
2. 复杂 OA 审批流。
3. 完整财务、合同、支付系统。
4. 高级 BI 报表系统。
5. 复杂权限组织架构。
6. 完整生产级 NL2SQL Agent。

## 2.3 权限原则

1. 学生只能访问自己的服务数据。
2. 员工只能访问自己负责的客户和日报。
3. 班主任只能访问自己负责学生的服务记录。
4. 领导查看汇总数据时，心理类数据应做脱敏处理。
5. 管理员可管理系统配置，但敏感 API Key 不应明文展示。
6. AI 生成内容不能绕过权限边界。

# 三、系统总体架构

## 3.1 总体架构

系统采用四层架构：

```text
前端 / Dify WebApp
        ↓
Dify 平台
        ↓ HTTP
FastAPI 后端服务
        ↓
MySQL / Redis / MinIO / Milvus
```

## 3.2 各层职责

| 层级 | 技术 | 职责 |
|---|---|---|
| 前端层 | Vue 3 / Dify WebApp | 页面展示、表单提交、聊天入口、报告展示 |
| AI 编排层 | Dify | 意图识别、RAG 检索、LLM 生成、工作流编排 |
| 业务服务层 | FastAPI | 业务接口、权限校验、数据读写、Dify 工具 API |
| 数据存储层 | MySQL / Redis / MinIO / Milvus | 业务数据、缓存、文件、向量数据 |

## 3.3 技术选型

| 技术 | 用途 | 说明 |
|---|---|---|
| FastAPI | 后端服务 | 提供 REST API、Dify 工具 API |
| SQLAlchemy ORM | 数据访问 | 管理 MySQL 表模型 |
| Pydantic V2 | 数据校验 | 请求体、响应体、配置管理 |
| MySQL | 业务数据库 | 存储用户、客户、学生、日报、报告等数据 |
| Redis | 缓存/会话/任务状态 | 支撑异步任务状态、会话缓存 |
| MinIO | 文件存储 | 存储上传的 PDF、Excel、简历等文件 |
| Milvus | 向量数据库 | 存储知识向量，支持 RAG 检索 |
| Dify | AI 平台 | 工作流、Agent、RAG、LLM 调用 |
| Vue 3 | 自建前端 | 后台管理页面 |
| Element Plus | UI 组件库 | 表单、表格、弹窗、菜单 |
| ECharts | 图表 | 报告可视化 |

# 四、功能模块需求

## 4.1 客户研判模块

### 4.1.1 模块目标

客户研判模块用于接收客户资料，结合企业已有画像规则，通过 AI 分析客户背景，判断其是否匹配对应产品线，并输出产品推荐、匹配分数和研判理由。

### 4.1.2 使用角色

| 角色 | 使用场景 |
|---|---|
| 员工 | 上传客户资料，查看研判结果 |
| 管理员 | 管理画像规则、查看全部研判记录 |
| Dify | 调用 FastAPI 工具接口或被 FastAPI 调用进行 AI 研判 |

### 4.1.3 输入数据

支持以下输入：

1. 文本内容。
2. PDF 简历。
3. Excel 客户表。
4. 手动录入客户资料。
5. 批量导入客户资料。

### 4.1.4 核心流程

```text
上传客户资料
→ 保存文件或文本
→ 创建 customer_source 记录
→ 返回 source_id 和 pending 状态
→ 后端异步解析资料
→ 读取画像规则
→ 调用 Dify 进行 AI 研判
→ 写入 customer_profile
→ 前端轮询查询结果
```

客户研判应采用异步流程，上传接口不等待 AI 完整结果，而是先返回任务 ID，后续通过查询接口获取状态和结果。

### 4.1.5 功能需求

| 编号 | 功能 | 说明 |
|---|---|---|
| CR-001 | 上传客户资料 | 支持文本、PDF、Excel、手动录入 |
| CR-002 | 创建研判任务 | 上传成功后生成 source_id |
| CR-003 | 文件解析 | 提取 PDF、Excel、文本内容 |
| CR-004 | 画像规则匹配 | 读取产品线规则进行匹配 |
| CR-005 | AI 研判 | 调用 Dify 生成结构化研判结果 |
| CR-006 | 查询研判状态 | 返回 pending、processing、success、failed |
| CR-007 | 查询研判结果 | 返回客户画像、匹配分数、推荐产品 |
| CR-008 | 失败记录 | AI 异常或解析失败时记录失败原因 |

### 4.1.6 状态定义

| 状态 | 含义 |
|---|---|
| pending | 已上传，等待处理 |
| processing | 正在解析或 AI 研判 |
| success | 研判成功 |
| failed | 研判失败 |

### 4.1.7 输出结果

```json
{
  "source_id": 1,
  "parse_status": "success",
  "customer_name": "张三",
  "contact_info": "138xxxx0001",
  "background_info": {
    "education": "本科",
    "target_country": ["英国", "澳大利亚"],
    "target_major": "计算机"
  },
  "match_result": "matched",
  "matched_product": "留学申请",
  "match_score": 85.5,
  "match_reason": "学历背景、目标国家和专业方向符合产品要求",
  "recommended_programs": []
}
```

### 4.1.8 验收标准

1. 能上传文本、PDF、Excel 至少三类资料。
2. 上传后立即返回 `source_id`。
3. 任务状态可查询。
4. 成功后返回结构化 JSON。
5. AI 输出异常时不导致系统崩溃。
6. 失败任务可查看错误原因。
7. 接口可通过 Swagger 或 Postman 测试。

## 4.2 客服 Agent 模块

### 4.2.1 模块目标

客服 Agent 面向客户和访客，支持公司信息、业务查询、政策查询、课程推荐、活动报名、FAQ、日常闲聊等 7 大场景。

### 4.2.2 使用角色

| 角色 | 使用场景 |
|---|---|
| 访客/客户 | 咨询服务、课程、活动、政策 |
| 员工 | 查看会话记录和客户线索 |
| Dify | 执行对话工作流、调用 FastAPI 接口 |

### 4.2.3 场景划分

| 场景 | 实现方式 | 是否调用 FastAPI |
|---|---|---|
| 公司信息 | Dify RAG 知识库 | 否 |
| 业务查询 | Dify RAG 知识库 | 可选 |
| 留学政策 | Dify RAG 知识库 | 否 |
| 课程推荐 | Dify HTTP 请求节点 | 是 |
| 活动报名 | Dify HTTP 请求节点 | 是 |
| FAQ | Dify RAG 知识库 | 否 |
| 闲聊 | LLM 直接回复 | 否 |

客服 Agent 在 Dify 中应配置知识库节点、意图识别节点、HTTP 请求节点，用于完成场景识别和业务接口调用。

### 4.2.4 功能需求

| 编号 | 功能 | 说明 |
|---|---|---|
| CS-001 | 创建会话 | 创建或获取访客/用户会话 |
| CS-002 | 保存聊天记录 | 保存用户问题、AI 回复、命中场景 |
| CS-003 | 查询课程列表 | 支持按分类、价格、标签筛选 |
| CS-004 | 查询活动列表 | 支持按状态、类型、时间筛选 |
| CS-005 | 活动报名 | 收集姓名、联系方式并报名 |
| CS-006 | FAQ 问答 | 基于 Dify 知识库回答 |
| CS-007 | 线索沉淀 | 将高意向访客转为 crm_lead |
| CS-008 | 人工转接 | 无法回答时提示转人工 |

### 4.2.5 活动报名规则

1. 活动必须存在。
2. 活动状态必须为可报名。
3. 当前报名人数不得超过最大人数。
4. 同一手机号不允许重复报名同一活动。
5. 报名成功后写入 `event_registration`。
6. 报名成功后 `event_lecture.current_participants` 加 1。
7. 并发报名最后一个名额时，只允许一个成功。

### 4.2.6 兜底规则

当用户问题超出业务范围时，Agent 不得编造课程、价格、政策，应回复：

```text
当前问题超出我的业务范围，建议联系人工顾问进一步确认。
```

### 4.2.7 验收标准

1. 用户可以通过客服 Agent 查询课程。
2. 用户可以查询活动并完成报名。
3. 课程和活动数据来自 FastAPI，而不是 AI 编造。
4. 会话记录可入库。
5. Dify HTTP 节点能成功调用 FastAPI。
6. 活动报名并发时不超额。

## 4.3 企业智能助手模块

### 4.3.1 模块目标

企业智能助手面向员工和管理层，支持意向客户管理、跟进记录、口述日报、日报汇总、组织架构查询、新人入职指引、学生业务综管等功能。

| 管理员 | 管理组织架构、用户、权限 |
| Dify | 识别自然语言意图，辅助结构化日报和查询 |

### 4.3.3 功能需求

| 编号 | 功能 | 说明 |
|---|---|---|
| EA-001 | 新增意向客户 | 录入客户姓名、联系方式、背景、负责人 |

| EA-003 | 更新客户状态 | 支持新增、跟进中、已签约、已流失 |
| EA-004 | 新增跟进记录 | 记录沟通方式、内容、下一步计划 |
| EA-005 | 查询跟进历史 | 查询某客户所有跟进记录 |
| EA-006 | 提交日报 | 支持文本或语音转文本后的日报提交 |
| EA-007 | AI 结构化日报 | Dify 将口述内容整理成结构化日报 |
| EA-008 | 日报汇总 | 按日期、部门、员工汇总日报 |
| EA-009 | 简单业务查询 | 通过预设模板查询客户或日报 |
| EA-010 | 组织架构查询 | 查询员工、部门、角色信息 |

### 4.3.4 客户状态规则

| 状态 | 说明 | 是否终态 |
|---|---|---|
| new | 新增意向 | 否 |
| contacting | 跟进中 | 否 |
| qualified | 有效客户 | 否 |
| signed | 已签约 | 是 |
| lost | 已流失 | 是 |

规则：

1. `signed` 和 `lost` 为终态，原则上不能回退到 `new`。
2. 变更为 `lost` 时必须填写流失原因。
3. 状态变更需要记录操作人和时间。
4. 客户必须有负责人。

### 4.3.5 NL2SQL 需求边界

本期不直接实现完整自由 NL2SQL，优先实现预设查询模板。

#### 一期能力

```text
用户说：查询张三的跟进记录
系统识别：客户跟进查询模板
执行：预设 SQL
返回：跟进记录列表
```

#### 二期能力

1. Dify 生成 SQL。
2. 后端校验 SQL。
3. 只允许 `SELECT`。
4. 限制可查询表。
5. 记录自然语言问题、生成 SQL、执行结果。
6. 查询结果交给 LLM 总结。

### 4.3.6 验收标准

1. 员工能新增、查询、更新客户。
2. 员工能新增跟进记录。
3. 员工能提交日报。
4. 领导能按日期查看日报汇总。
5. 预设自然语言查询能返回正确业务数据。
6. 非授权员工不能查看他人客户明细。

## 4.4 学生智能助手模块

### 4.4.1 模块目标

学生智能助手面向在读学生和服务老师，支持请假审批、心理关怀预警、投诉反馈、学业考务查询、申请进度追踪、海外生活支持、增值转化等场景。

### 4.4.2 使用角色

| 角色 | 使用场景 |
|---|---|
| 学生 | 请假、投诉、查成绩、查申请进度 |
| 班主任/服务老师 | 审批请假、处理预警、处理投诉 |
| 管理员 | 查看全局服务数据 |
| Dify | 识别学生意图、情绪和服务场景 |

### 4.4.3 功能需求

| 编号 | 功能 | 说明 |
|---|---|---|
| SA-001 | 提交请假申请 | 学生提交请假类型、时间、原因 |
| SA-002 | 审批请假 | 班主任审批通过或驳回 |
| SA-003 | 查询请假记录 | 学生查询本人请假记录 |
| SA-004 | 记录情绪交互 | 保存 Dify 识别出的情绪标签和分值 |
| SA-005 | 心理预警生成 | 根据情绪分值和关键词触发预警 |
| SA-006 | 查询心理预警 | 班主任查看待处理预警 |
| SA-007 | 处理心理预警 | 记录跟进情况并更新状态 |
| SA-008 | 提交投诉/建议 | 学生提交投诉工单 |
| SA-009 | 处理投诉 | 老师或员工处理并反馈 |
| SA-010 | 查询申请进度 | 查询学校申请、签证、材料状态 |
| SA-011 | 查询 DDL | 查询未来 N 天截止事项 |
| SA-012 | 学业考务查询 | 查询成绩、考试、课程安排 |

### 4.4.4 请假审批流程

```text
学生提交请假
→ 状态 pending
→ 写入待办
→ 班主任审批
→ 状态 approved / rejected
→ 写入通知记录
→ 学生查看结果
```

### 4.4.5 心理预警流程

```text
学生与 AI 对话
→ Dify 识别情绪标签、分值、关键词
→ 调用 FastAPI 保存心理记录
→ 后端判断风险等级
→ 生成心理预警
→ 推送给班主任
→ 班主任跟进
→ 更新预警状态
```

### 4.4.6 心理预警等级

| 等级 | 触发条件 | 处理要求 |
|---|---|---|
| 轻度 | 情绪分值偏低，无高危关键词 | 记录情绪，建议关怀 |
| 中度 | 多次负面情绪，出现明显压力表达 | 创建待办，班主任跟进 |
| 重度 | 出现自伤、自杀、绝望等高危表达 | 立即预警，人工介入 |

重要约束：

1. AI 只做风险识别，不做医学诊断。
2. 高危预警必须进入人工处理流程。
3. 心理数据必须权限隔离。
4. 心理风险阈值应保守，宁可多报，不可漏报。

### 4.4.7 投诉处理规则

1. 学生可提交投诉或建议。
2. 投诉必须生成工单编号。
3. 工单状态包括：待处理、处理中、已解决、已关闭。
4. 超过 3 天未处理，应提升优先级。
5. 处理完成后应记录解决方案。
6. 学生可查看本人投诉处理进度。

### 4.4.8 验收标准

1. 学生能提交请假申请。
2. 班主任能审批请假。
3. 学生能提交投诉。
4. 投诉工单能被处理并返回结果。
5. 情绪低分或高危关键词能触发预警。
6. 班主任能处理心理预警。
7. 学生能查询申请进度和 DDL。

## 4.5 智能报告模块

### 4.5.1 模块目标

智能报告模块用于汇总业务数据，并调用 Dify 生成结构化分析报告，服务管理层决策。

### 4.5.2 报告类型

| 报告类型 | 数据来源 | 输出内容 |
|---|---|---|
| 全域客户经营分析 | crm_lead、crm_follow_up | 客户来源、转化率、流失原因 |
| 员工日报汇总 | employee_daily_report | 工作进展、风险、计划 |
| 学生心理周报 | student_psych_record、student_psych_alert | 情绪趋势、风险人数、处理状态 |
| 投诉处理周报 | student_feedback_ticket | 投诉分类、处理时效、高频问题 |

### 4.5.3 报告生成流程

```text
创建报告任务
→ 写入 report_generation，status=generating
→ 聚合业务数据
→ 组装 Prompt 和结构化数据
→ 调用 Dify 生成报告
→ 保存 report_content 和 report_html
→ 更新 status=completed
```

### 4.5.4 功能需求

| 编号 | 功能 | 说明 |
|---|---|---|
| RP-001 | 触发报告生成 | 手动触发指定类型报告 |
| RP-002 | 定时生成报告 | 支持日/周定时任务 |
| RP-003 | 查询报告列表 | 按类型、状态、时间查询 |
| RP-004 | 查看报告详情 | 返回结构化内容和 HTML |
| RP-005 | 报告失败记录 | 保存失败原因 |
| RP-006 | 报告重新生成 | 对失败或过期报告重新生成 |

### 4.5.5 报告状态

| 状态 | 含义 |
|---|---|
| generating | 生成中 |
| completed | 已完成 |
| failed | 失败 |
| expired | 已过期 |

### 4.5.6 验收标准

1. 能手动触发报告生成。
2. 报告生成过程不阻塞主接口。
3. 报告生成完成后可查询详情。
4. 报告内容包含结构化 JSON 和 HTML。
5. Dify 调用失败时记录失败原因。
6. 至少完成一种周报演示。

| 实体 | 说明 |

## 5.2 数据设计原则

1. 所有核心表必须包含主键。
2. 关键业务表应包含创建时间和更新时间。
3. 状态字段应使用枚举值，不应随意填写中文自由文本。
4. AI 输出结果应同时保存原始内容和结构化结果。
5. 敏感数据需脱敏展示。
6. 文件只保存 URL 或对象存储路径，不直接存入数据库。
7. 大字段内容如报告正文、AI 结果，可使用 JSON 或 TEXT。

## 5.3 数据安全要求

1. 用户密码必须加密存储，不允许明文。
2. API Key 不允许写入代码仓库。
3. 心理记录和预警属于敏感数据。
4. 投诉内容仅授权人员可见。
5. 日志中不得打印完整 API Key、密码、身份证号等敏感信息。

# 九、非功能性需求

## 9.1 性能需求

| 类型 | 要求 |
|---|---|
| 普通查询接口 | 1 秒内返回 |
| 普通写入接口 | 2 秒内返回 |
| AI 研判 | 采用异步任务，不阻塞上传接口 |
| 报告生成 | 采用异步任务，不阻塞用户操作 |
| 活动报名 | 支持并发控制，不超卖 |

## 9.2 安全需求

1. 密码加密存储。
2. API Key 存入 `.env`。
3. 前端不暴露 Dify 后台管理 Key。
4. 学生心理数据和投诉数据权限隔离。
5. 日志脱敏。
6. 文件上传限制类型和大小。
7. SQL 查询必须防注入。
8. NL2SQL 只允许 SELECT。

## 9.3 可用性需求

1. Dify 调用失败时，业务接口应返回明确错误。
2. 数据库异常时应记录日志。
3. 文件解析失败时任务状态变为 `failed`。
4. AI 输出非 JSON 时应进入异常处理。
5. 报告生成失败后可重新生成。

## 9.4 可维护性需求

1. 后端按 routers、schemas、services、repositories、models 分层。
2. 业务逻辑不得堆在 router 层。
3. 配置统一通过 settings 管理。
4. 日志统一封装。
5. 接口统一返回格式。
6. 数据库表字段命名统一。

## 9.5 日志需求

至少记录：

1. 接口请求入口。
2. 关键业务状态变化。
3. Dify 请求和响应摘要。
4. 异步任务状态。
5. 异常堆栈。
6. 用户操作记录。

不得记录：

1. 完整 API Key。
2. 明文密码。
3. 高敏感心理原文。
4. 大段无用日志。

# 十四、结论

本项目的核心不是“做一个聊天机器人”，而是建设一个 **AI 编排 + 业务系统 + 数据闭环** 的教育服务平台。

工程落地时应坚持以下原则：

```text
先业务闭环，再 AI 增强；
先接口稳定，再工作流编排；
先数据入库，再智能分析；
先 MVP 演示，再生产增强。
```

优先完成一条可稳定演示的主流程：

```text
用户提问
→ Dify 识别意图
→ 调用 FastAPI
→ 查询 MySQL
→ 返回业务数据
→ Dify 组织自然语言回复
→ 前端展示
```

这条主链路跑通后，再扩展客户研判、学生心理预警、智能报告和 NL2SQL。
