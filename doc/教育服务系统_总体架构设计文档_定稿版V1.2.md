# 1. 文档说明

## 1.1 文档目标

本文档用于描述教育服务系统在 MVP 阶段的总体架构设计，明确系统分层、模块职责、数据流、接口规范、部署方式、安全策略和后续扩展方向。

本版是对原 V1.1 架构文档的全链路评审定稿版，重点进行了以下调整：

1. 明确 3 天 MVP 开发范围。
2. 收敛过重的生产级设计。
3. 统一 API 路径规范。
4. 统一数据库访问方式。
5. 明确 Dify 与 FastAPI 权限边界。
6. 明确异步任务状态落库机制。
7. 明确 Redis、MinIO、Milvus、Celery 等组件为后续增强能力。
8. 修正“4 层架构”与实际图示不一致的问题。
9. 对齐数据库 V2.1、API V1.2、Dify V1.1、前端 V1.1 的版本基线。
10. 明确 P0 演示以 Dify WebApp + Swagger 为兜底，Vue 后台按 P1 逐步补齐。

## 1.2 项目定位

项目名称：**教育服务系统（留学服务智能平台）**

项目定位：基于 **FastAPI + Dify + MySQL** 的 AI 驱动教育服务系统。

系统面向留学教育服务机构，覆盖以下核心场景：

1. 客户咨询与客服 Agent。
2. 课程查询与活动报名。
3. 企业员工 CRM 客户管理。
4. 学生请假、投诉等服务闭环。
5. 智能报告生成。
6. 客户研判、心理预警、日报汇总、NL2SQL 等能力按 MVP 取舍分阶段实现；其中客户研判属于业务核心模块，但 3 天 MVP 可先作为 P1 验证链路。

## 1.3 核心架构比喻

```text
Dify    = 大脑：负责意图识别、对话生成、RAG 检索、工作流编排
FastAPI = 手脚：负责业务接口、数据读写、权限校验、并发控制
MySQL   = 记忆：负责业务数据持久化，是唯一真实数据源
前端    = 操作面板：负责页面展示、表单提交、状态查看
```

对应非标设备类比：

| 软件架构组件 | 非标设备类比 | 说明 |
|---|---|---|
| Dify | 上位机/智能调度系统 | 判断用户意图，决定调用哪个业务接口 |
| FastAPI | PLC + 执行机构 | 执行业务动作，完成数据读写 |
| MySQL | 物料仓/数据台账 | 保存真实业务数据 |
| Vue / Dify WebApp | HMI 操作屏 | 用户操作入口 |
| API 接口 | 气路/电气接口 | 连接模块之间的数据流 |

# 2. MVP 架构范围

## 2.1 MVP 必须启用组件

3 天开发周期内，必须启用以下组件：

| 组件 | 是否必须 | 用途 |
|---|---|---|
| FastAPI | 必须 | 后端业务接口 |
| MySQL | 必须 | 核心业务数据存储 |
| Dify | 必须 | AI 对话与工作流编排 |
| Swagger | 必须 | 接口测试与答辩展示 |
| Dify WebApp | 必须 | 快速演示 Agent 能力 |
| Vue 3 前端 | 可选但建议 | 后台管理页面展示 |
| 本地文件目录 | 可选 | 替代 MinIO，用于 MVP 文件上传 |

## 2.2 MVP 暂不启用组件

以下组件保留设计位置，但不作为 3 天 MVP 必做项：

| 组件 | MVP 阶段处理方式 | 后续增强方式 |
|---|---|---|
| Redis | 暂不启用 | 用于缓存、任务状态、队列 |
| Celery | 暂不启用 | 替代 BackgroundTasks 做可靠异步任务 |
| MinIO | 可用本地 uploads 替代 | 正式文件对象存储 |
| Milvus | 使用 Dify 内置知识库替代 | 自建向量检索 |
| Alembic | 先使用 db_init.sql | 表结构稳定后引入迁移管理 |
| 完整 RBAC | MVP 简化角色校验 | 后续做完整权限模型 |
| 完整 NL2SQL | 先做模板查询 | 后续做 LLM SQL 生成与校验 |

## 2.3 MVP 必须跑通的主链路

```text
用户在 Dify WebApp 提问
        ↓
Dify 识别意图
        ↓
Dify HTTP 节点调用 FastAPI
        ↓
FastAPI 查询 MySQL
        ↓
FastAPI 返回业务 JSON
        ↓
Dify 基于接口结果组织回复
        ↓
用户看到自然语言结果
```

MVP 阶段优先保证这条链路稳定。其他模块都围绕这条主链路扩展。

# 3. 总体架构设计

## 3.1 架构分层

本系统采用 **4 层架构 + 2 个横切能力**。

> 说明：原 V1 文档中将“网关层”单独列出，但实际 MVP 阶段没有独立 API Gateway，因此本版将认证、CORS、统一响应、日志等能力归入 FastAPI 业务服务层的横切能力中。

```text
┌───────────────────────────────────────────────────────────────┐
│  1. 接入层 Presentation Layer                                  │
│                                                               │
│  Vue 3 前端 / Dify WebApp / Swagger / Postman                  │
└───────────────────────────┬───────────────────────────────────┘
                            │
┌───────────────────────────▼───────────────────────────────────┐
│  2. AI 编排层 AI Orchestration Layer                           │
│                                                               │
│  Dify Chatflow / Workflow / Agent                              │
│  - 意图识别                                                     │
│  - 对话生成                                                     │
│  - RAG 知识库问答                                                │
│  - HTTP 节点调用 FastAPI                                        │
└───────────────────────────┬───────────────────────────────────┘
                            │
┌───────────────────────────▼───────────────────────────────────┐
│  3. 业务服务层 Business Service Layer — FastAPI                 │
│                                                               │
│  routers / schemas / services / models / utils                 │
│  - 业务接口                                                     │
│  - 数据校验                                                     │
│  - 权限校验                                                     │
│  - 并发控制                                                     │
│  - Dify API 调用封装                                             │
│  - 统一响应、日志、异常处理                                      │
└───────────────────────────┬───────────────────────────────────┘
                            │
┌───────────────────────────▼───────────────────────────────────┐
│  4. 数据存储层 Data Storage Layer                               │
│                                                               │
│  MySQL 8.0                                                     │
│  - 用户数据                                                     │
│  - 课程活动数据                                                  │
│  - CRM 数据                                                     │
│  - 学生服务数据                                                  │
│  - 报告数据                                                     │
└───────────────────────────────────────────────────────────────┘

横切能力 1：统一规范
- API 前缀：/api/v1
- 统一响应：{code, message, data}
- 统一错误码
- 命名规范
- 状态枚举

横切能力 2：安全与可观测
- JWT 或 MVP 简化鉴权
- Dify Service Token
- 日志脱敏
- 请求日志
- Dify 调用摘要日志
- 异常统一处理
```

## 3.2 各层职责

| 层 | 职责 | 技术 | 禁止事项 |
|---|---|---|---|
| 接入层 | 页面展示、用户输入、接口调用 | Vue 3、Dify WebApp、Swagger | 不直接操作数据库 |
| AI 编排层 | 意图识别、对话生成、RAG、工作流 | Dify | 不做业务数据持久化 |
| 业务服务层 | CRUD、权限、事务、并发、接口封装 | FastAPI、SQLAlchemy | 不编造 AI 业务数据 |
| 数据存储层 | 保存真实业务数据 | MySQL | 不承载业务逻辑 |

## 3.3 核心模块划分

| 模块 | 代码标识 | MVP 范围 |
|---|---|---|
| 基础设施 | auth / health / common | 健康检查、简化认证、统一响应 |
| 客服 Agent | course / event / chat | 课程查询、活动查询、活动报名 |
| 企业助手 | crm / employee | 意向客户 CRUD、跟进记录 |
| 学生助手 | student | 请假或投诉闭环 |
| 智能报告 | report | 报告生成与查询 |
| Dify 集成 | dify_service | FastAPI 调 Dify、Dify 调 FastAPI |

# 4. Dify 与 FastAPI 职责边界

## 4.1 核心职责划分

| 能力 | Dify 负责 | FastAPI 负责 |
|---|---|---|
| 意图识别 | 是 | 否 |
| 自然语言回复 | 是 | 否 |
| RAG 知识库问答 | 是 | 否 |
| 业务数据查询 | 否 | 是 |
| 数据新增/修改/删除 | 否 | 是 |
| 活动报名并发控制 | 否 | 是 |
| 权限校验 | 辅助 | 是 |
| 状态流转 | 否 | 是 |
| 报告文本生成 | 是 | 可辅助 |
| 报告数据聚合 | 否 | 是 |
| 业务日志 | 否 | 是 |
| 数据库访问 | 否 | 是 |

## 4.2 必须遵守的边界

1. **Dify 不直接操作数据库。**
2. **Dify 不保存业务主数据。**
3. **AI 不编造课程、活动、客户、学生等业务数据。**
4. **所有真实业务数据必须来自 FastAPI 查询 MySQL 的结果。**
5. **所有写入动作必须经过 FastAPI。**
6. **活动报名、状态变更、审批等业务规则必须在 FastAPI 中校验。**

## 4.3 Dify 可调用接口白名单

MVP 阶段 Dify 只允许调用以下 FastAPI 接口：

| 方法 | 路径 | 场景 |
|---|---|---|
| GET | `/api/v1/courses` | 课程查询 |
| GET | `/api/v1/events` | 活动查询 |
| POST | `/api/v1/events/{event_id}/register` | 活动报名 |
| GET | `/api/v1/student/applications` | 申请进度查询，P1 |
| POST | `/api/v1/student/psych/record` | 心理记录写入，P1 |
| GET | `/api/v1/crm/leads` | 客户查询，P1 |
| POST | `/api/v1/chat/session` | 会话记录，P1 |

不允许 Dify 直接调用以下接口：

1. 用户管理接口。
2. 系统配置接口。
3. 删除类接口。
4. 管理员接口。
5. 未列入白名单的写入接口。

## 4.4 Dify Service Token 设计

Dify 调用 FastAPI 时使用独立服务令牌：

```text
Authorization: Bearer {DIFY_SERVICE_TOKEN}
```

该 Token 与用户登录 JWT 区分。

设计目的：

1. 区分用户请求与 Dify 服务请求。
2. 便于日志审计。
3. 限制 Dify 可调用接口范围。
4. 避免 Dify 获得过高系统权限。

FastAPI 侧建议实现独立依赖：

```python
from fastapi import Header, HTTPException
from config import settings


def verify_dify_service_token(authorization: str = Header(...)) -> None:
    if not authorization.startswith("Bearer "):
        raise HTTPException(status_code=403, detail="Invalid service token")

    token = authorization.replace("Bearer ", "").strip()
    if token != settings.DIFY_SERVICE_TOKEN:
        raise HTTPException(status_code=403, detail="Invalid service token")
```

# 5. 核心数据流设计

## 5.1 用户登录链路

MVP 阶段可采用简化登录，也可实现 JWT 登录。推荐接口统一为：

```text
POST /api/v1/auth/login
GET  /api/v1/auth/me
```

链路：

```text
前端提交用户名密码
        ↓
FastAPI 查询 sys_user
        ↓
校验密码
        ↓
生成 Token
        ↓
返回用户信息和 Token
        ↓
前端保存 Token
        ↓
后续请求携带 Authorization Header
```

注意：所有文档和代码必须统一使用 `/api/v1/auth/login`，不得混用 `/auth/login`。

## 5.2 客服课程查询链路

```text
用户输入：“有什么语言培训课程？”
        ↓
Dify 意图识别：课程查询
        ↓
Dify HTTP 节点调用：
GET /api/v1/courses?category=语言培训
        ↓
FastAPI 查询 course_project 表
        ↓
返回课程 JSON
        ↓
Dify 根据 JSON 组织自然语言回复
```

关键约束：

1. Dify 只基于接口返回的数据回复。
2. 如果课程为空，回复“暂无相关课程”。
3. 课程价格、时间、课程名不得由 AI 编造。

## 5.3 活动报名链路

```text
用户查询活动
        ↓
Dify 返回活动列表
        ↓
用户选择活动并提供姓名、手机号
        ↓
Dify 调用：
POST /api/v1/events/{event_id}/register
        ↓
FastAPI 开启事务
        ↓
检查活动是否存在、是否发布、是否还有名额、是否重复报名
        ↓
写入 event_registration
        ↓
更新 event_lecture.current_participants
        ↓
返回报名成功
```

并发控制策略：

1. MVP 推荐使用数据库事务。
2. 如果使用 MySQL InnoDB，可采用 `SELECT ... FOR UPDATE`。
3. 必须保证报名人数不超过 `max_participants`。
4. 失败时返回明确错误码。

## 5.4 客户研判异步链路

MVP 可先实现文本资料研判，PDF / Excel 可作为 P1。

```text
员工提交客户资料
        ↓
FastAPI 创建 customer_source 记录，status=pending
        ↓
立即返回 source_id
        ↓
BackgroundTasks 后台执行
        ↓
读取资料和规则
        ↓
调用 Dify Chat API
        ↓
解析 AI 输出 JSON
        ↓
写入 customer_profile
        ↓
更新状态 success / failed
        ↓
前端轮询查询结果
```

重要说明：

```text
接口层面：异步，不阻塞用户请求
后台任务内部：MVP 可以使用 Dify blocking 模式获取完整结果
后续增强：升级为 Celery + Redis，支持重试、进度追踪和多实例部署
```

## 5.5 智能报告生成链路

```text
管理员触发报告生成
        ↓
FastAPI 创建 report_generation，status=generating
        ↓
后台任务聚合 CRM / 学生 / 日报数据
        ↓
调用 Dify 生成报告文本
        ↓
保存 report_content / report_html
        ↓
更新 status=completed
        ↓
前端查询报告详情
```

报告生成状态：

| 状态 | 含义 |
|---|---|
| pending | 待生成 |
| generating | 生成中 |
| completed | 已完成 |
| failed | 失败 |

# 6. 数据库架构设计

## 6.1 数据库选型

MVP 阶段使用：

```text
MySQL 8.0
字符集：utf8mb4
排序规则：utf8mb4_unicode_ci
```

## 6.2 数据库访问方式

为了降低 3 天开发风险，MVP 阶段统一采用：

```text
SQLAlchemy 同步模式 + PyMySQL
```

推荐连接串：

```text
mysql+pymysql://root:password@localhost:3306/education_service?charset=utf8mb4
```

不建议 MVP 阶段使用异步 `AsyncSession + aiomysql`，原因：

1. 新手容易混用同步/异步写法。
2. `db.query()` 与 `AsyncSession` 不兼容。
3. 3 天开发周期下排错成本较高。
4. 同步 SQLAlchemy 已足够支撑 MVP 演示。

后续 P2 阶段可升级为：

```text
SQLAlchemy AsyncSession + aiomysql
```

## 6.3 核心表分组

| 分组 | 核心表 | MVP |
|---|---|---|
| 基础认证 | sys_user、sys_role | P0 |
| 客服 Agent | course_project、event_lecture、event_registration | P0 |
| 对话记录 | chat_session、chat_message | P1 |
| 企业助手 | crm_lead、crm_follow_up、employee_daily_report | P0/P1 |
| 学生助手 | student_admin_service、student_feedback_ticket | P0 |
| 心理预警 | student_psych_profile、student_psych_record、student_psych_alert | P1 |
| 智能报告 | report_generation | P0 |
| 客户研判 | customer_source、customer_profile、profile_rule | P1 |

## 6.4 核心 ER 关系

```text
sys_user ──1:N── crm_lead
crm_lead ──1:N── crm_follow_up

sys_user ──1:N── student_admin_service
sys_user ──1:N── student_feedback_ticket

event_lecture ──1:N── event_registration

report_generation ──N:1── sys_user

customer_source ──1:1── customer_profile
```

## 6.5 表设计原则

1. 表名使用小写下划线。
2. 字段名使用小写下划线。
3. 主键统一为 `id`。
4. 时间字段统一为 `created_at`、`updated_at`。
5. 状态字段使用英文枚举。
6. 文件只保存路径，不直接存入数据库。
7. AI 输出结果保存原始文本和结构化 JSON。
8. 关键写入接口必须记录操作人。

```text
│
```

## 7.2 分层调用规则

```text
routers  → services → models / database
schemas  → routers / services
utils    → 被各层调用
```

禁止规则：

1. Router 不直接写复杂业务逻辑。
2. Router 不直接写复杂 SQL。
3. Service 不调用 Router。
4. B/C/D 业务模块不直接互相调用。
5. 跨模块聚合统一由 report_service 通过数据库查询完成。

## 7.4 Dify 调用封装

FastAPI 调 Dify 时统一封装在 `services/dify_service.py`。

MVP 阶段建议：

1. 使用 `httpx`。
2. `response_mode` 使用 `blocking`。
3. 外层由 BackgroundTasks 异步触发。
4. 捕获异常并更新任务状态为 `failed`。

示例：

```python
import httpx
from config import settings


class DifyService:
    def __init__(self):
        self.base_url = settings.DIFY_API_BASE_URL
        self.api_key = settings.DIFY_API_KEY

    def chat(self, query: str, user: str, inputs: dict | None = None) -> dict:
        payload = {
            "inputs": inputs or {},
            "query": query,
            "response_mode": "blocking",
            "user": user,
        }

        with httpx.Client(timeout=30.0) as client:
            response = client.post(
                f"{self.base_url}/chat-messages",
                headers={
                    "Authorization": f"Bearer {self.api_key}",
                    "Content-Type": "application/json",
                },
                json=payload,
            )
            response.raise_for_status()
            return response.json()
```

# 10. 安全架构设计

## 10.1 MVP 安全策略

| 安全点 | MVP 方案 | 后续增强 |
|---|---|---|
| 用户认证 | 简化 JWT 或模拟用户 | 完整 JWT + Refresh Token |
| Dify 鉴权 | DIFY_SERVICE_TOKEN | 服务账号 + 接口白名单 |
| 密码存储 | bcrypt 哈希 | 增加密码策略 |
| 敏感配置 | `.env` | 密钥管理系统 |
| 日志脱敏 | 不打印密码、Token、API Key | 结构化日志审计 |
| SQL 注入 | SQLAlchemy 参数化查询 | SQL 审计 |
| 心理数据 | 权限隔离 | 更细粒度授权和脱敏 |

## 10.2 心理数据安全边界

如果实现学生心理预警，必须遵守：

1. AI 只做风险识别和提醒。
2. AI 不做医学诊断。
3. 高危预警必须进入人工处理流程。
4. 心理记录仅授权角色可见。
5. 日志中不记录完整心理原文。
6. 答辩演示数据必须使用模拟数据。

## 10.3 日志脱敏规则

敏感字段：

```text
password
password_hash
token
access_token
api_key
secret
id_card
phone
psych_content
```

处理策略：

1. 密码、Token、API Key 全部替换为 `***`。
2. 手机号只显示后四位。
3. 心理内容不写入普通日志。
4. Dify 调用日志只记录摘要和状态，不记录完整敏感内容。

# 12. 架构决策记录 ADR

## ADR-001：后端框架选择 FastAPI

| 项 | 内容 |
|---|---|
| 状态 | 已采纳 |
| 决策 | 使用 FastAPI |
| 理由 | 自动 Swagger、类型友好、适合 API 项目、开发效率高 |
| 备选 | Django、Flask |
| 影响 | 团队需按 FastAPI 分层规范开发 |

## ADR-002：MVP 数据库访问选择同步 SQLAlchemy

| 项 | 内容 |
|---|---|
| 状态 | 已采纳 |
| 决策 | MVP 使用 SQLAlchemy 同步模式 + PyMySQL |
| 理由 | 降低 3 天开发和排错成本 |
| 备选 | SQLAlchemy AsyncSession + aiomysql |
| 影响 | 后续高并发场景可升级异步模式 |

## ADR-003：Dify 与 FastAPI 双向调用

| 项 | 内容 |
|---|---|
| 状态 | 已采纳 |
| 决策 | Dify 可调用 FastAPI 工具接口，FastAPI 也可调用 Dify Chat API |
| 理由 | 同时支持对话型业务和后台 AI 任务 |
| 影响 | 必须明确调用白名单和 Token 权限 |

## ADR-004：异步任务 MVP 使用 BackgroundTasks

| 项 | 内容 |
|---|---|
| 状态 | MVP 阶段采纳 |
| 决策 | 客户研判、报告生成使用 FastAPI BackgroundTasks |
| 理由 | 简单、无额外组件、适合演示 |
| 限制 | 不支持持久化队列、重试、多实例 |
| 后续 | 升级 Celery + Redis |

## ADR-005：文件存储 MVP 使用本地 uploads

| 项 | 内容 |
|---|---|
| 状态 | MVP 阶段采纳 |
| 决策 | 先使用本地目录保存上传文件 |
| 理由 | 避免 3 天内接入 MinIO 增加复杂度 |
| 后续 | 升级 MinIO 对象存储 |

## ADR-006：RAG MVP 使用 Dify 内置知识库

| 项 | 内容 |
|---|---|
| 状态 | MVP 阶段采纳 |
| 决策 | 不单独接 Milvus，优先使用 Dify 内置知识库 |
| 理由 | 降低部署和联调成本 |
| 后续 | 数据规模扩大后再接 Milvus |

# 14. 关键风险与控制措施

| 风险 | 说明 | 控制措施 |
|---|---|---|
| 架构过重 | 3 天内无法完成全部组件 | 明确 MVP 裁剪 |
| Dify 调接口失败 | Docker 网络或 URL 错误 | 使用 host.docker.internal |
| 数据库异步混用 | AsyncSession 与 db.query 混用 | MVP 统一同步 SQLAlchemy |
| AI 编造业务数据 | LLM 自行生成课程或活动 | Prompt 强约束，只基于接口返回 |
| 活动报名超额 | 并发报名最后名额 | 使用事务和 FOR UPDATE |
| BackgroundTasks 丢任务 | 服务重启任务丢失 | 任务状态落库，失败可重试 |
| 心理数据合规风险 | 敏感数据暴露 | 模拟数据、权限隔离、AI 不诊断 |
| 前端进度不足 | 页面做不完 | Dify WebApp + Swagger 兜底 |
| 答辩演示失败 | 网络或服务异常 | 准备截图、录屏和备用数据 |

