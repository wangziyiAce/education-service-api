## 一、整体架构：注册表驱动的报告编排系统

在进入十份报告之前，必须先把**架构**讲清楚。这个模块经历了一次从 V1 到 V2 的重构，核心思想是：

> **"把一份报告从「一笔糊涂账」拆成注册、聚合、规则、AI、渲染五个独立环节"**

### 1.1 架构分层图

```text
┌──────────────────────────────────────────────────────────┐
│                      触发来源                              │
│  手动 POST /generate  │  定时 APScheduler  │  重试 retry  │
└───────────────────────┬──────────────────────────────────┘
                        ▼
┌──────────────────────────────────────────────────────────┐
│               Orchestrator（编排层）                        │
│  · create_report_task() → 创建任务 + 幂等检查              │
│  · generate_report() → 同步执行全流程                      │
│  · retry_report() → 新记录，不覆盖原记录                    │
└───────────┬──────────────┬──────────────┬────────────────┘
            ▼              ▼              ▼
┌───────────────┐ ┌──────────────┐ ┌──────────────────────┐
│  Aggregators  │ │   Rules      │ │  AI Generator        │
│  聚合器        │ │   规则引擎    │ │  (Dify Chatflow)     │
│               │ │              │ │                      │
│ SQL取数+聚合   │ │ 纯函数计算：  │ │ 只允许改写：          │
│ 数字来源：     │ │ · 风险分     │ │ · summary            │
│ 数据库事实表   │ │ · ROI/CPL    │ │ · explanation        │
│               │ │ · SLA超时    │ │ 禁止改写任何业务数字！  │
└───────┬───────┘ └──────┬───────┘ └──────────┬───────────┘
        │                │                    │
        └────────────────┼────────────────────┘
                         ▼
              ┌──────────────────┐
              │   Registry        │
              │   注册表           │
              │  report_type →    │
              │  aggregator_name  │
              │  content_model    │
              │  template_name    │
              │  allowed_roles    │
              │  default_period   │
              └──────────────────┘
                         ▼
              ┌──────────────────┐
              │   Renderer        │
              │   HTML模板渲染     │
              └──────────────────┘
```

### 1.2 设计亮点一：数字与解释分离

这是模块最核心的设计理念：

| 职责 | 谁来做 | 为什么 |
|------|--------|--------|
| 业务数字（风险分、转化率、ROI） | SQL + `rules.py` | 必须可追溯、可验证、可审计 |
| 自然语言解释（摘要、建议） | Dify Chatflow | AI 只做"翻译"，不做"编造" |
| HTML 渲染 | `renderer.py` + 后端模板 | 安全，不信任模型输出的 HTML |

### 1.3 设计亮点二：注册表驱动扩展

新增一份报告只需三步，不用改任何路由代码：

1. 在 `registry.py` 加一条 `_definition(...)`
2. 在 `schemas.py` 定义一个 `XxxContent(BaseReportContent)` Pydantic 模型
3. 在 `aggregators.py` 写一个 `aggregate_xxx()` 函数

### 1.4 设计亮点三：三层报告嵌套

报告不是平级的，而是有三层嵌套关系：

```text
第三层（综合报告）: weekly_summary
  ├── 第二层（复用报告）: customer_ops, psych_weekly, complaint_weekly
  │     ├── 第一层（原子报告）: sales_funnel → customer_ops 复用
  │     ├── 第一层（原子报告）: service_sla → psych_weekly / complaint_weekly 复用
  │     └── 第一层（原子报告）: daily_summary → psych_weekly 也复用
  └── 第一层（独立报告）: application_risk, channel_roi, action_closure
```

**面试可以这样讲：**
> "综合周报是一个报告之上的报告——它不重复取数，而是组合已有报告的结果。销售漏斗、服务 SLA 等原子报告被多份上层报告复用，体现了 DRY 原则和模块化设计。"

### 第 1 份：客户经营分析报告 (`customer_ops`)

**解决什么业务问题：** 管理者想知道"我的客户池子里有多少人、在哪个阶段、哪些人快流失了"。

**数据来源：** `crm_lead` 表（通过 `aggregate_sales_funnel` 复用取数逻辑）。

**核心指标：**
- 总线索数、签约率
- 阶段分布（new → contacting → qualified → signed → contracted）
- 停滞线索列表（超过 N 天未跟进）
- 流失分析

**数据流：**

```text
crm_lead 表
  → aggregate_sales_funnel() 统计漏斗阶段 + 停滞客户
  → aggregate_customer_ops() 复用漏斗结果，补充流失分析
  → CustomerOpsContent Schema 校验
  → Dify 补充 summary / explanation
  → 渲染 customer_ops.html 模板
```

**设计亮点：** 客户经营报告复用了销售漏斗的聚合逻辑（`aggregate_customer_ops` 内部调用 `aggregate_sales_funnel`），体现了**"一个事实源可以被多份报告复用"**的设计思想。

**前端渲染：** 无专用渲染器 → 进入 `GenericV2Renderer`，展示指标卡片 + 阶段分布表格 + 停滞线索表格 + AI 摘要/解释。

---

### 第 2 份：员工日报汇总报告 (`daily_summary`)

**解决什么业务问题：** 团队管理者想知道"我的团队这周做了什么、有什么风险、下一步计划是什么"。

**数据来源：** `employee_daily_report` 表。

**核心指标：**
- 提交日报份数
- 提交率
- 关键进展汇总
- 共性风险
- 下一步计划

**数据流：**

```text
employee_daily_report 表
  → COUNT(*) 统计提交数
  → 报告仅做数量统计，不做内容分析（保护隐私）
  → DailySummaryContent Schema 校验
  → Dify 补充自然语言说明
  → 渲染 daily_summary.html 模板
```

**默认周期：** 上一个自然日（`previous_day`），适合每日早晨自动生成。

**设计亮点：** 日报汇总只做数量统计，不做内容分析——保护员工隐私。对比很多系统直接把日报全文塞给 AI，这里选择"只统计不阅读"的隐私友好设计。

---

### 第 3 份：综合经营周报 (`weekly_summary`)

**解决什么业务问题：** 高层管理者不想看四份独立报告，想要一个"飞机仪表盘"式的全景视图。

**数据来源：** 聚合了 `customer_ops` + `daily_summary` + `psych_weekly` + `complaint_weekly` 四份报告的结果。

**核心指标：**
- `business_sections`：四份子报告的完整内容
- `cross_module_risks`：跨模块的交叉风险
- `management_actions`：管理层建议行动

**数据流：**

```text
aggregate_weekly_summary()
  ├── aggregate_customer_ops()  → CRM 模块数据
  ├── aggregate_daily_summary() → 日报模块数据
  ├── aggregate_psych_weekly()  → 心理预警模块数据
  └── aggregate_complaint_weekly() → 投诉模块数据
  → 汇总到 WeeklySummaryContent.business_sections
  → Dify 跨模块解读
  → 渲染 weekly_summary.html 模板
```

**设计亮点：** 这是典型的**"组合模式"**——不重复取数，而是组合已有报告的结果。面试时可以讲："综合周报是一个报告之上的报告，它展示了跨模块的数据关联能力，而不只是简单的拼接。"

**权限：** 仅限 `admin` 和 `manager`，因为包含心理预警等敏感数据。

---

### 第 4 份：学生心理预警周报 (`psych_weekly`)

**解决什么业务问题：** 心理辅导团队需要追踪"有多少学生在预警、什么等级、有没有人超过 1 小时还没有跟进"。

**数据来源：** `student_psych_alert` 表。

**核心指标：**
- 预警总条数、高风险预警数量
- 情绪趋势（时间序列）
- 预警状态分布（pending / following / resolved）
- **高风险跟进时效：是否超过 1 小时未跟进**

**核心隐私保护规则：**

> "心理报告只统计趋势、等级、状态和时效，禁止输出学生原文或诊断。"

这意味着：
- `aggregated_data_snapshot` 不保存学生原文
- Dify 的 `inputs` 不传入学生原文
- 聚合 SQL 只查询 `risk_level`、`status`、`first_follow_time`，不查询诊断内容

**数据流：**

```text
student_psych_alert 表
  → 查询 risk_level, status, first_follow_time, create_time
  → 高风险 + 超过 1 小时未跟进 → 标记超时
  → PsychWeeklyContent Schema 校验
  → Dify 只收到聚合数据，不收到学生原文
  → 渲染 psych_weekly.html 模板
```

**权限：** `admin`, `manager`, `team_leader`（普通员工看不到心理数据）。

**前端渲染：** 有专用渲染器 `PsychWeeklyRenderer`，展示：
- 顶部隐私保护声明（红色横幅）
- 风险等级分布卡片
- 预警状态表格（已匿名，只显示风险等级、处理状态、首次响应小时数）
- 首次跟进时效卡片
- AI 分析摘要（仅匿名统计）

---

### 第 5 份：投诉处理周报 (`complaint_weekly`)

**解决什么业务问题：** 服务质量团队需要知道"投诉处理得快不快、有没有超时、哪些问题反复出现"。

**数据来源：** `student_feedback_ticket` 表。

**核心指标：**
- 投诉总量
- 首次响应超时数和解决超时数
- 平均首次响应时间
- SLA 明细（每单的实际耗时 vs 阈值）
- 高发问题分类

**SLA 阈值规则（来自 `rules.py`）：**

| 优先级 | 首次响应时限 | 解决时限 |
|--------|-------------|---------|
| urgent | 1小时 | 24小时 |
| high | 4小时 | 48小时 |
| medium | 24小时 | 72小时 |
| low | 48小时 | 120小时 |

**数据流：**

```text
student_feedback_ticket 表
  → 查询 priority, create_time, first_response_time, resolved_time, status
  → evaluate_complaint_sla() 按优先级阈值判断每单是否超时
  → 汇编成 complaint_sla 明细列表
  → ComplaintWeeklyContent Schema 校验
  → Dify 补充高频问题分析
  → 渲染 complaint_weekly.html 模板
```

**设计亮点：** SLA 阈值是可配置的字典（`COMPLAINT_SLA_THRESHOLDS`），不是硬编码在 SQL 里的魔法数字。如果业务规则变化，改一处即可。

### 第 6 份：申请风险报告 (`application_risk`) ★ V2 新增

**解决什么业务问题：** 留学申请过程中，每个学生要提交大量材料。管理者不可能逐个人工核查。这份报告自动识别"哪些申请快逾期、哪些材料缺失、该联系谁"。

**数据来源：** `application_material_item` 表（V2 新增的事实表）。

**核心指标：**
- 高风险/中风险/低风险申请数量
- 逾期申请数
- 缺失材料总数
- 每个高风险申请的详细信息

**风险评分规则（`calculate_application_risk`）：**

| 触发条件 | 加分 |
|---------|------|
| 已逾期 | +40 |
| 7 天内截止 | +30 |
| 8-30 天截止 | +15 |
| 每项缺失必填材料 | +20（上限 40） |
| 7 天未更新 | +20 |
| 无下一步动作 | +10 |

**等级判定：** ≥70 高风险、40-69 中风险、<40 低风险。

**数据流：**

```text
application_material_item 表
  → 按 application_id 分组聚合
  → 对每个申请调用 calculate_application_risk() 评分
  → 过滤出 risk_score >= 40 的写入 risk_items
  → ApplicationRiskContent Schema 校验
  → Dify 生成 action_checklist 建议
  → 渲染 application_risk.html 模板
```

**设计亮点：**
- 规则完全可追溯：每个风险分都有 `risk_reasons` 字段记录触发了哪些规则
- 输出 `action_checklist`：不只是"告诉你有风险"，而是"告诉谁该做什么、什么时候做完"
- 面试时可以重点讲："这个风险分不是 AI 拍脑袋给的，每 1 分都有规则来源，可以直接回溯到申请材料表的截止日期和提交状态"

**前端渲染：** 有专用渲染器 `ApplicationRiskRenderer`，展示：
- 6 个风险指标卡片（总申请、高/中/低风险、逾期数、缺材料数）
- 风险明细表格（申请ID、阶段、风险分、风险等级、风险原因、缺失材料、下一步）
- 行动建议列表（带优先级和截止日）
- AI 分析摘要 + 解释面板

### 第 7 份：销售漏斗报告 (`sales_funnel`)

**解决什么业务问题：** 销售管理者想知道"这个月进了多少线索、漏了多少、卡在哪个阶段、哪个人业绩好"。

**数据来源：** `crm_lead` 表。

**核心指标：**
- 漏斗各阶段数量（new → contacting → qualified → signed）
- 转化率
- 各阶段平均停留天数
- 停滞线索（不同阶段有不同停滞阈值）
- 顾问业绩排行

**停滞判断规则：**

| 阶段 | 阈值 | 风险等级 |
|------|------|---------|
| new | ≥1 天未跟进 | high |
| contacting | ≥3 天未跟进 | medium |
| qualified | ≥2 天未跟进 | high |

**数据流：**

```text
crm_lead 表
  → GROUP BY status 统计漏斗数量
  → 按 last_follow_time 判断停滞
  → calculate_conversion_rate(signed, total)
  → SalesFunnelContent Schema 校验
  → Dify 补充销售策略建议
  → 渲染 sales_funnel.html 模板
```

**设计亮点：** 转化率使用**同一 Cohort（同一创建周期）**计算，而不是混合不同批次的客户。面试时可以说："我选择的 Cohort 方法能避免'上个月的线索和这个月的签约'被错误配对，让转化率反映真实销售效率。"

### 第 8 份：渠道 ROI 报告 (`channel_roi`) ★ V2 新增

**解决什么业务问题：** 市场投放花了多少钱、带来了多少线索、签了多少单、实际回款多少、每个渠道的投资回报率是多少。

**数据来源：** 四张表联合取数：
- `marketing_channel_cost` → 投放成本
- `crm_lead` → 线索数
- `customer_contract` → 签约数和合同金额
- `customer_payment` → 实际回款金额

**核心公式（来自 `rules.py`）：**

```text
CPL（每条线索成本）= 渠道成本 / 线索数
CAC（获客成本）= 渠道成本 / 签约数
ROI = (实际回款 - 渠道成本) / 渠道成本
```

**数据流：**

```text
四张表并行取数：
  ├── marketing_channel_cost → costs dict
  ├── crm_lead → leads dict
  ├── customer_contract → signed dict (count + amount)
  └── customer_payment → paid dict

  → 以 channel 为 key 合并四张表
  → calculate_channel_roi() 计算 CPL/CAC/ROI
  → ChannelROIContent Schema 校验
  → Dify 补充渠道优化建议
  → 渲染 channel_roi.html 模板
```

**设计亮点：**
- 使用 `Decimal` 做金额计算，避免浮点数精度问题（`0.3000000004` 这种尴尬）
- 成本为 0 时返回 `None` 而非 0——"None 表示数据不足，0 表示真实转化为 0，两者业务含义不同"
- JOIN 了 `customer_payment` 和 `customer_contract`，用**实际回款**而非合同金额算 ROI，更反映真实现金流

**权限：** 仅限 `admin` 和 `manager`（渠道成本是商业敏感数据）。

**前端渲染：** 有专用渲染器 `ChannelRoiRenderer`，展示：
- 渠道指标表格（渠道名、成本、线索数、CPL、签约数、CAC、回款、ROI）
- ROI 正/负值用不同颜色图标区分
- ROI 为 null 时显示"数据不足"标签
- 数据质量警告区域

### 第 9 份：服务 SLA 报告 (`service_sla`) ★ V2 新增

**解决什么业务问题：** 服务质量部门需要一份统一的"服务水平协议达标率"视图，覆盖投诉、行政服务和心理预警三个服务线。

**数据来源：** 两张表：
- `student_feedback_ticket` → 投诉工单
- `student_psych_alert` → 心理预警

**核心指标：**
- 投诉首次响应超时数 / 解决超时数
- 平均首次响应小时数
- 心理预警高风险跟进超时数
- 各服务线 backlog 积压情况

**数据流：**

```text
两张表并行取数：
  ├── student_feedback_ticket → complaint_sla 列表
  │   → evaluate_complaint_sla() 逐单判断
  └── student_psych_alert → psych_alert_sla 列表
      → 高风险 + 超过1小时未跟进 → 标记超时

  → 汇总 sla_overview（跨服务线总览）
  → ServiceSLAContent Schema 校验
  → Dify 补充服务质量趋势分析
  → 渲染 service_sla.html 模板
```

**设计亮点：** SLA 报告把两条服务线放到一张报告里对比，能让管理者发现"投诉响应很快但心理预警跟进很慢"这种跨部门问题。

### 第 10 份：报告行动闭环报告 (`action_closure`) ★ V2 新增

**解决什么业务问题：** AI 给的建议不能停留在报告里，必须有人跟进执行。这份报告追踪"建议 → 行动 → 完成"的闭环。

**数据来源：** `report_action` 表（V2 新增）。

**核心指标：**
- 行动项总数
- 完成率 = done / total
- 按时完成率 = 按时 done / total
- 逾期未完成项列表
- 重复出现的问题（同一 `risk_code` 出现 ≥2 次）
- 目标达成情况

**数据流：**

```text
report_action 表
  → COUNT + GROUP BY status → 完成率
  → 过滤 due_time < now AND status != 'done' → 逾期项
  → GROUP BY risk_code → 重复问题识别
  → 过滤 target_value IS NOT NULL → 目标达成
  → ActionClosureContent Schema 校验
  → Dify 补充闭环改进建议
  → 渲染 action_closure.html 模板
```

**设计亮点：**
- **"AI 建议不会自动成为任务，必须进入 `report_action` 才参与闭环统计"**——这条规则写在注释和代码里，避免了 AI 建议满天飞但没人执行的窘境
- `risk_code` 字段让相同问题可以被归类，从而发现"同一个问题反复出现"的系统性风险
- `target_value` vs `actual_value` —— 行动不只要完成，还要验证效果

## 四、共有的技术设计

### 4.1 数据质量体系 (`DataQuality`)

每份报告都附带数据质量标记，比 V1 的"报告要么成功要么失败"细腻得多：

| 级别 | 含义 | 示例 |
|------|------|------|
| `ok` | 所有数据源正常 | - |
| `warning` | 部分可选数据源缺失 | 渠道成本表为空，ROI 标记为 None |
| `empty` | 统计周期内无数据 | 本周期没有 CRM 线索 |
| `degraded` | 降级生成 | 仅用部分数据源 |
| `failed` | 数据源完全不可用 | 必需表不存在 |

前端可以据此展示"报告已生成，但数据不完整"的半透明状态。

### 4.2 指标追溯 (`MetricTrace`)

每份报告都包含 `metric_traces` 数组，记录每个关键指标：
- 来自哪些表（`source_tables`）
- 计算公式是什么（`formula`）
- 过滤条件是什么（`filters`）

面试时被问"这个数字怎么来的"，可以直接指给面试官看。

**示例（渠道 ROI 的 MetricTrace）：**

```json
{
  "metric_name": "channel_roi",
  "source_tables": ["marketing_channel_cost", "crm_lead", "customer_contract", "customer_payment"],
  "formula": "CPL=成本/线索数，CAC=成本/签约数，ROI=(回款-成本)/成本",
  "filters": {"period_start": "2026-06-30", "period_end": "2026-07-06"}
}
```

### 4.3 幂等机制 (`idempotency_key`)

- 手动接口：取请求头 `Idempotency-Key`
- 定时任务：`schedule:{schedule_id}:{period_start}:{period_end}`
- 同一个 key 不会创建重复任务——这对定时调度多 worker 场景至关重要

### 4.4 三级权限控制

- **接口级**：学生角色无法访问 `/reports` 任何接口
- **报告类型级**：`channel_roi` 和 `weekly_summary` 限制为 `admin + manager`
- **记录级**：普通员工只能查看自己生成的报告

### 4.5 Schema 版本兼容

`ReportEnvelope` 的 `model_validator` 只在 `schema_version >= 2` 时做严格的 Pydantic 模型校验，V1 旧数据保持 dict 不强制迁移——向后兼容但不阻塞前进。

### 4.6 前端三级渲染策略

```
V2 专用渲染器 (3个重点报告)
  → GenericV2Renderer (Schema V2 通用渲染)
    → GenericV1Renderer (V1 历史数据兼容)
```

### 4.7 独立定时调度进程

```text
API 服务 (多个 worker)          APScheduler 独立进程
       │                              │
       │                              ├── 读取 report_schedule 表
       │                              ├── cron 触发 → run_schedule_once()
       │                              ├── 调用 orchestrator.create_report_task()
       │                              └── 通过幂等键防止重复
       │
       └── 手动 POST /generate → 同样的 orchestrator 入口
```

**为什么不用 FastAPI 进程直接跑定时任务？** 在真实项目中，API 服务可能启动多个 worker。如果每个 worker 都启动调度器，同一份周报可能被生成多次。独立 APScheduler 进程配合幂等键可以避免重复。

## 五、完整数据流（以"申请风险报告"为例）

这是最完整的端到端流程（7 步），其他报告基本同理：

```text
═══════════════════════════════════════════════════════════════
Step 1: 触发
═══════════════════════════════════════════════════════════════
  POST /api/v1/reports/generate
  Body: {"report_type": "application_risk", "report_title": "7月申请风险周报"}
  Header: Idempotency-Key: abc123

═══════════════════════════════════════════════════════════════
Step 2: 权限校验
═══════════════════════════════════════════════════════════════
  get_current_user()
  → ensure_report_permission(user, ("admin","manager","employee","team_leader"))
  → 学生角色直接 403

═══════════════════════════════════════════════════════════════
Step 3: 创建任务记录 (create_report_task)
═══════════════════════════════════════════════════════════════
  ├── get_report_definition("application_risk") → 查注册表
  ├── resolve_period() → 解析统计周期（默认 previous_week）
  ├── build_idempotency_key() → "manual:abc123"
  ├── 查重：是否已有相同 idempotency_key 的任务？有则直接返回已有记录
  └── INSERT INTO report_generation (status='pending', ...)

═══════════════════════════════════════════════════════════════
Step 4: 返回 202 Accepted
═══════════════════════════════════════════════════════════════
  前端立即拿到 task_id，不用等报告生成完成
  后台通过 FastAPI BackgroundTasks 异步执行

═══════════════════════════════════════════════════════════════
Step 5: 后台异步执行 (generate_report_async)
═══════════════════════════════════════════════════════════════
  ├── 创建独立 db session（不能复用请求 session，请求可能已关闭）
  └── 调用 generate_report(report_id, db)

═══════════════════════════════════════════════════════════════
Step 6: generate_report() 主流程
═══════════════════════════════════════════════════════════════

  6a. 状态变更
      status → 'generating'
      started_time → now

  6b. 数据聚合 (aggregate_application_risk)
      ├── 查 application_material_item 表
      ├── 按 application_id 分组
      ├── 对每个申请调用 calculate_application_risk()
      │   ├── overdue? +40
      │   ├── days_to_deadline ≤ 7? +30
      │   ├── days_to_deadline 8-30? +15
      │   ├── missing_required_materials × 20 (max 40)
      │   ├── days_since_update ≥ 7? +20
      │   └── no next_action? +10
      ├── 分级：≥70 high, ≥40 medium, <40 low
      ├── 过滤 risk_score ≥ 40 写入 risk_items
      ├── 生成 action_checklist（每个高风险项一条行动建议）
      └── 生成 metric_traces（追溯信息）

  6c. AI 解释层 (enrich_content_with_ai)
      ├── 检查 REPORT_AI_MODE 环境变量
      ├── 如果 != "dify"：走 _local_explanation() 本地模式
      │   └── 明确标记 data_source = "local"，不是伪造 AI
      ├── 如果 = "dify"：
      │   ├── 把聚合数据 + expected_schema + data_quality 打入 inputs
      │   ├── POST Dify /chat-messages（timeout=180s）
      │   ├── 解析响应 JSON（支持 markdown code block 格式）
      │   ├── 只取 summary 和 explanation 字段
      │   ├── Pydantic Schema 校验
      │   │   └── 失败 → 再调一次 Dify 修复（带上 validation_error）
      │   │       └── 仍失败 → 抛出异常，任务进入 failed
      │   └── 返回合并后的 content（业务数字仍是聚合器的结果）

  6d. Schema 再校验
      ApplicationRiskContent.model_validate(content) ← 再次确认结构正确

  6e. HTML 渲染 (render_report_html)
      └── 使用 application_risk.html 模板
          └── 如果模板不存在 → 使用内置 fallback 模板

  6f. 持久化
      ├── report_content = content_dict (完整结构化内容)
      ├── report_html = 渲染后的 HTML 字符串
      ├── aggregated_data_snapshot = 聚合快照（不含 AI 解释）
      ├── data_quality = 数据质量标记
      └── status → 'completed'

═══════════════════════════════════════════════════════════════
Step 7: 前端获取
═══════════════════════════════════════════════════════════════
  GET /api/v1/reports/{id}
  → 返回完整 ReportDetailResponse（含 report_content + report_html）
  → 前端根据 report_type + schema_version 选择渲染器
  → ApplicationRiskRenderer 渲染专用风险组件
```

## 六、面试表达指南

### 6.1 如何概括整个模块

**面试官问："你做的智能报告模块是怎么设计的？"**

> 我把智能报告模块从单文件 MVP 重构成了一个**注册表驱动的报告编排系统**，核心设计原则是**数字与解释分离**。
>
> 业务数字（风险分、转化率、ROI、SLA 超时）全部由 SQL 和规则引擎在后端计算，每一条都可以追溯到数据库事实表和计算公式；大模型只负责把数字翻译成人能看懂的摘要和建议。
>
> 模块分成六个独立层：注册表管理十种报告类型的元数据、聚合器负责从事实表取数、规则引擎做纯函数计算、AI 层调用 Dify Chatflow 补充解释、渲染层用后端模板生成 HTML、编排层管理任务状态和幂等。
>
> 新增报告类型只需要在注册表加一条配置、定义一个 Pydantic Schema、写一个聚合函数，不用改任何路由代码。
>
> 我还实现了数据质量分级、指标追溯、幂等去重、Schema 版本兼容、三级权限控制和独立的 APScheduler 定时调度进程。

### 6.2 如果追问"数字可追溯"是什么意思

> 报告中每一个指标都附带 `metric_traces` 字段，记录它来自哪些表、计算公式是什么、用了什么过滤条件。比如渠道 ROI，我知道它是从 `marketing_channel_cost`、`crm_lead`、`customer_contract`、`customer_payment` 四张表 JOIN 出来的，公式是 `(回款 - 成本) / 成本`。即使半年后回头看报告，也能明确知道每个数字是怎么来的。

### 6.3 如果追问"为什么不让大模型直接输出报告"

> 两个原因。第一，安全和可靠——大模型可能编造数字，而管理报告的数字必须准确，否则影响决策。第二，前端体验——模型输出的 HTML 样式不可控，后端模板渲染能保证所有报告样式一致。
>
> 我采用的做法是：后端把结构化 JSON 传给 Dify Chatflow，Dify 只能改写 `summary` 和 `explanation` 字段。代码里用 Pydantic 校验确保模型不会越界改写业务数字。即使 Dify 不可用，系统也能用本地模板模式生成报告。

### 6.4 如果追问"重试是怎么设计的"

> 失败不覆盖原记录。调用 `POST /{id}/retry` 会创建一个**新的记录**，通过 `retry_of_report_id` 指向原来的失败记录，`retry_count` 递增。这样做的原因是：保留失败现场，方便排查；同时防止重试覆盖掉原始错误信息。
>
> 定时任务的重复保护通过 `idempotency_key = schedule:{schedule_id}:{period_start}:{period_end}` 实现。如果调度进程因为某种原因在同一周期内被触发两次，第二次会命中已有任务，不会重复生成。

### 6.5 如果追问"申请风险分的规则为什么这样设计"

> 规则来自业务方确认的需求，不是我自己凭感觉定的。每条规则都有明确的业务理由：
>
> - 已逾期 +40：因为一旦逾期，整个申请流程可能中断，这是最严重的风险
> - 截止日 7 天内 +30：学生和顾问只有一周时间准备，紧急程度很高
> - 缺失材料每项 +20（上限 40）：材料补齐需要时间，但不会像逾期那样致命
> - 7 天未更新 +20：表示负责人可能忘记跟进，属于管理风险
> - 无下一步动作 +10：流程停滞但没有那么紧急
>
> 这个规则是可插拔的——如果业务方未来想调整权重或加新规则，只需要改 `calculate_application_risk` 这个纯函数，所有调用方自动生效。而且因为这个函数是纯函数，我写了完整的单元测试覆盖所有边界情况。
